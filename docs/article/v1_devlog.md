# Проектируем AI-агента вместе с Claude: от идеи до архитектуры

*Не «как использовать LLM», а как выглядит настоящий co-design на практике.*

---

## Глава 0 — Почему пишем своё, когда вокруг уже всё есть

Первый вопрос разумного человека: «есть Letta, OpenHands, LangGraph,
smolagents, aider, Continue.dev — зачем ещё одно?». Мы честно полезли
проверять — research-агенту в фоне, по техническим докам, лицензиям и
зависимостям. Вывод оказался полезным сам по себе.

**Не подходит:**

- **Letta** (бывший MemGPT) хочет владеть **всем циклом агента**:
  свой runtime, Postgres + pgvector, обязательная embedding-модель,
  Docker-first. Memory не отделяется от движка — это сервер, не
  библиотека. Lock-in несовместим с принципом «маленькое и локальное».
- **OpenHands agent-sdk** — полноценный coding agent. Композируется
  лучше Letta, но всё равно хочет владеть Conversation/Environment.
  Полезен **как источник идей** (особенно `FileEditorTool` —
  концептуально близко к нашему SEARCH/REPLACE), но не как
  dependency.
- **LangChain / LangGraph** — тяжёлые, тащат много мусора, и для
  одного-агентного линейного цикла избыточны. LangGraph пригодится
  только если будем делать branching state-machine, чего пока нет.
- **AutoGen** — multi-agent. Мы single-agent. Wrong abstraction.
- **aider** — best-in-class CLI для weak-LLM patch-flow. Мы у них
  **уже взяли** самое ценное — SEARCH/REPLACE формат. Сам aider —
  CLI, не библиотека, не для embedding.
- **Continue.dev** — TypeScript, IDE-плагин. Не наш стек, не наш UX.

**Возможно стоит — в v0.3:**

- **mem0** (`pip install mem0ai`, Apache-2.0) — узкий слой памяти,
  не агент. API из двух методов: `add(text)` / `search(query)`.
  Под капотом — LiteLLM (поддерживает LM Studio), local Chroma,
  семантический retrieval, дедупликация. Лёгкие зависимости,
  никакого Postgres. Закрыл бы ~30% v0.3 memory pillar (recall и
  persistence). Остальные 70% — AST индексация, symbol graph,
  summaries — пишем сами, потому что **никто не делает это хорошо
  для слабых моделей**. Это и есть наш moat.

**Возможно стоит — в v0.4+:**

- **smolagents** (HF, Apache-2.0) — лёгкий, philosophy «code-as-action»:
  агент пишет Python вместо JSON tool calls. Конфликтует с нашей
  ставкой на native function-calling для слабых моделей, но интересен
  для iterative patch loop через год.
- **LangGraph** — когда понадобится branching (planner-architect-
  implementer split).

**Главное наблюдение:**

Все «готовые» агенты разделяются на два класса. Либо **они хотят
владеть всем** (Letta, OpenHands) — и тогда инструмент уже не твой,
ты живёшь в их runtime. Либо **они компонуемые** (mem0, smolagents) —
но закрывают только узкий слой, и собирать из них агента всё равно
тебе.

Для weak-LLM кейса где **половина инженерии — это контекст** (что
модель видит в каждом turn'е: project map, docstrings, imports,
tool-результаты), готовые решения мало что дают. Они оптимизированы
под frontier-модели где «правильный промт + достаточно контекста»
хватит. У нас другая задача: компенсировать слабость модели
**структурой данных**, не количеством параметров.

Вывод: **пишем сами, mem0 спайкнем в v0.3 memory layer**, AST/symbol
graph/summaries оставляем себе как core competency.

## Откуда идея

Хотел инструмент для работы с кодом через слабые локальные модели — qwen2.5-coder-14b в LM Studio. Не замену Claude Code, а что-то скромное: маленькая задача, минимальный патч, тест, дальше. Назвал `code-scalpel`.

Вместо того чтобы сразу писать код — решил сначала спроектировать в диалоге с Claude. Получилось интереснее чем ожидал.

## Как выглядел процесс

Не «скажи что делать» и не «сделай за меня». Скорее: я приношу идею, Claude уточняет, оба критикуем, решение кристаллизуется.

Первый вопрос от Claude был не про стек, а про ограничения: какая целевая модель, что точно не нужно в MVP. Это определяет дизайн сильнее чем список фич.

Результат первой сессии — `plan.md` на 35 секций. Не код, а документ: стек, режимы, форматы файлов, примеры STATE.json, роадмап. Потом отдал план на ревью другой модели. Внешний взгляд нашёл реальные проблемы.

## Решения которые пришли через спор

**i18n в v0.1.** Казалось разумным — сразу на двух языках. Claude сам же предложил убрать: «зачем, если главная гипотеза — работает ли qwen14b на реальном проекте». Убрали в v0.4.

**review vs ask.** Два режима с нечёткой границей. После нескольких итераций пришли к одному критерию: `ask` — диалог, полные файлы не читаются; `review` — читает код, выдаёт структурированный отчёт. Внешняя модель предложила третий режим `inspect` как промежуточный — я отклонил. Три похожих режима сложнее одного чёткого разреза.

**LLM-summaries при init.** Первый вариант: `init` прогоняет каждый файл через LLM для INDEX. На локальной 14B это минуты. Исправление: по умолчанию AST-символы без LLM, LLM только при `--summarize`.

## Ключевые архитектурные решения

### Stateless между шагами

Главный инсайт: никакой истории разговора между LLM-вызовами. Каждый вызов — свежий промт:

```
[system.md] + [mode.md] + [stable_context] + [assistant: OK] + [dynamic_context]
```

`stable_context` меняется редко — ARCH.md, INDEX metadata, skill snippets. Сидит в KV-кэше.
`dynamic_context` меняется каждый шаг — задача, файлы, traceback, diff.

Вместо истории — step summaries в TASKS.md. После каждого шага: 1–3 строки что было сделано. Следующий шаг читает summary, не полную историю. Это и есть встроенный compact.

### Контекстный бюджет как первоклассная концепция

Не «постараемся уложиться», а явные лимиты и стратегии сжатия по приоритету:

```
1. file index      → только релевантные записи
2. code snippets   → обрезать до N строк + список символов
3. test output     → только traceback
4. git diff        → без context lines
→ всё равно не влезло → StopCondition(context_budget_exceeded)
```

`context_tokens` — автодетект из `/v1/models` API. Никаких хардкодных чисел. Все пороги (когда бар желтеет, когда предлагать compact) — в конфиге.

### Compact как семантический переход

Compact не только про «контекст кончается». Второй сценарий: переходишь из `ask` в `plan` после долгого обсуждения. Тащить весь чат не нужно — нужны решения. Агент суммаризирует диалог, суммари идёт в контекст планировщика. Планировщик не переспрашивает то что уже решили.

### Skills как Markdown-плагины

Новый язык или инструмент — не нужно менять код приложения. Создаёшь `.md` с YAML frontmatter:

```yaml
---
name: typescript
type: language
file_patterns: ["*.ts", "*.tsx"]
test_cmd: ["npx", "jest"]
symbol_pattern: "^(?:export\\s+)?(?:class|function)\\s+(\\w+)"
---
```

Агент сканирует директории при старте, приоритет: project > user > builtin. Команда `learn` генерирует skill через LLM из знаний модели или URL документации.

## TUI: от экранов к потоку

Первый вариант: отдельные Textual Screen-ы для каждой фазы. PatchScreen, PlanScreen, TestScreen. Это неправильно — ломает ощущение непрерывности.

Правильно: один экран, бесконечный скролл, inline-карточки в потоке. Вдохновение — Claude Code. Всё идёт в одну ленту, карточки появляются где нужно и становятся read-only после действия. Никаких попапов.

Начали с 8 типов карточек. После анализа что реально отличается — осталось 4:

| Тип | Что делает |
|---|---|
| `ToolCallCard` | все вызовы инструментов: read, search, git, apply, tests |
| `PlanCard` | список задач с прогрессом |
| `FormCard` | любая форма: QA-уточнения, настройки |
| `ChoiceCard` | бинарный выбор: resume, compact-offer |

`ToolCallCard` для Apply имеет два состояния: интерактивный (diff + [a]/[r]/[g]) и read-only (результат). Один виджет, меньше когнитивной нагрузки.

## Какую боль решаем

Рынок AI coding tools выглядит примерно так:

| Сегмент | Что есть |
|---|---|
| «магия в облаке» | Claude Code, Cursor, GitHub Copilot |
| «автономные агенты» | OpenHands, OpenDevin |
| «patch assistant» | Aider |
| «terminal + local» | куски через Ollama, LM Studio |

Почти нет инструмента с философией: локальная модель + маленький контекст + patch-oriented + supervised autonomy + TUI-first + engineering discipline. Это и есть ниша code-scalpel.

Но важнее — конкретная боль. У людей есть 7B–14B модели, но нет нормального coding workflow под них. Типичный опыт сейчас:

```
OpenCode/OpenHands
↓ контекст взорвался
↓ модель галлюцинирует
↓ агент переписал полпроекта
↓ всё сломалось
```

code-scalpel строит противоположный UX: не доверяй модели, ограничивай её, режь scope, показывай patch, тестируй быстро.

Аудитория — Linux/devops люди, локальные AI-хоббисты, privacy-first пользователи, embedded/infra/backend разработчики, те кто ненавидит web IDE, те у кого нет RTX 5090.

**Идеологическая цельность** — отдельный актив. Многие AI tools выглядят так: добавим MCP, добавим browser, добавим swarm, добавим memory graph. Философия code-scalpel простая: точечные изменения под слабые модели. Для open-source проекта это важно.

Главный враг сейчас — feature creep. Если v0.1 реально заработает (patch loop + qwen14b + pytest + diff preview) — проект уже имеет ценность сам по себе.

---

## Что вынес

**Документ перед кодом.** `plan.md` — не документация к коду, а инструмент мышления. Хорошо написанный план находит противоречия до того как они войдут в реализацию.

**Внешнее ревью работает.** Отдать план другой модели — не проверка грамматики, а реально другой угол зрения. Нашли то что пропустили вдвоём.

**Scope discipline — самое сложное.** Не «что добавить», а «что убрать». i18n в v0.1 казалась разумной пока не задали вопрос: зачем это для проверки главной гипотезы?

**AI не замена мышлению.** Claude предлагал, я отклонял, оба искали лучшее решение. Полезнее всего — когда модель задаёт неудобные вопросы, а не просто соглашается.

## Глава 2 — формат diff'а

v0.1 поставили вопрос ребром: **умеет ли qwen2.5-coder-14b стабильно
выдавать применимый patch?** Сначала ответ был «да вроде» — один раз
запустили `poc_test.py`, всё сработало, поехали дальше. Но «вроде» —
не ответ.

Сделали бенчмарк из 15 типовых правок (тайпхинты, переименовать
функцию, добавить параметр, обработать пустой вход, заменить `.format`
на f-string, и т.п.). Завернули в `pytest --run-llm -m llm` — по
умолчанию пропускаются, запускаются по требованию. Каждый кейс
изолирован в `tmp_path` с собственным git-репо.

Первый прогон: **12/15 = 80%**. Стабильно между запусками. Все три
отказа — `git apply failed`, не «модель не выдала diff». То есть
**семантически модель понимает всё**, ломается на формате.

Каталог отказов:

1. `@@ -0,0 +N @@` на существующий файл — модель выбрала «формат для
   нового файла», когда хотела вставить строку в начало
2. Hunk без обязательного контекста (начинается со строки 2, пропускает
   `class User:` на строке 1) — git не находит точку приложения
3. Context-линия показывает **будущее** состояние, а не текущее —
   модель «дописала» сигнатуру функции в контекст с пробелом, потеряв
   возможность это «новое» отделить от старого

Все три — проблема не qwen в частности, а unified diff как формата
для LLM. Счётчики строк, hunk-заголовки, разделение контекста и
изменений — слишком много инвариантов для модели, которая «пишет код,
а не diff».

Параллельно проверили **спекулятивную декодировку** (LM Studio её
включает по умолчанию для qwen): отключили — те же 12/15, но в 1.5
раза медленнее. Сменили draft-модель на меньшую — ещё +10% скорости,
качество то же. Декодинг был не виновен.

Добавили fuzzy-fallback в applier — toleration к трейлинговым пустым
context-линиям. Дало 13/15. Дальше unified diff не вытянуть без
heuristic'ов, которые легко ломают валидные случаи.

Тут полезла другая мысль: **формат — выбор, а не данность**. Если
unified diff неудобен моделям, почему бы не сменить? Загуглили — и
обнаружили что [aider](https://github.com/Aider-AI/aider) уже три года
решает ту же задачу со слабыми моделями и пришёл к собственному
формату — `SEARCH/REPLACE blocks`:

```
file.py
<<<<<<< SEARCH
<точное содержимое которое надо найти>
=======
<чем заменить>
>>>>>>> REPLACE
```

Никаких счётчиков, никаких hunk-заголовков. Модель просто пишет
«вот что есть» и «вот что должно стать». aider к этому шёл
итерациями, поделил весь свой стек на эту задачу — мы получили
готовое решение бесплатно (Apache 2.0 совместима с AGPL, добавили в
`CREDITS.md`, код не копировали — повторили спецификацию своими
руками).

Урок: **формат, в котором модель общается с инструментом, — рычаг
сильнее чем размер модели и точность промта**. Если бенчмарк указывает
на формат как на узкое место — менять формат, а не строить
heuristic'и поверх.

Урок номер два: **бенчмарк превращает обсуждения в цифры**. Раньше
было «качество приемлемое, наверное». Теперь — «80% на unified diff,
13/15 после fuzzy, посмотрим что даст SEARCH/REPLACE». На каждое
изменение агента/промта прогон бенча даёт регрессии или улучшения
немедленно. Без него мы бы переключали форматы наобум.

## Глава 3 — контекст как ресурс

После v0.1 главный вопрос был «работает ли вообще». В v0.2 — «во
сколько это обходится». Цифра неприятная: на каждое сообщение,
включая «привет», уходило 5-8k токенов — мы eager-стафили в контекст
список файлов и первые три файла целиком. На большом проекте это
потолок.

Решение — два движения навстречу друг другу:

1. **Static project map** (`code_scalpel/project_map.py`). AST
   разбирает все Python-файлы в проекте и выдаёт компактную карту:
   путь, число строк, классы и top-level функции с сигнатурами.
   Малозначимые файлы — одной строкой. Кешируется в
   `.code-scalpel/INDEX.json` по mtime, регенерируется только то
   что менялось.

2. **Tool-calling loop**. Раньше мы дёргали `list_files` и `read_file`
   за модель — она получала результаты в одном жирном user-сообщении.
   Теперь модель сама вызывает инструменты: `<TOOL: read_file>`,
   `<TOOL: grep>`, `<TOOL: run_tests>`. Каждый ответ возвращается
   `<RESULT>` блоком, агент крутит цикл до ответа без вызовов.

Эффект: casual «привет» = ~200 токенов вместо 7k. Реальная правка
читает много файлов, но **только те которые нужны**.

Параллельно решили UX-задачу про tool-результаты. Если рендерить
`<RESULT>` в чат как есть — содержимое файла (по 300 строк) забивает
экран. Решение подсмотрено снова у Claude Code: каждый tool-call
становится **сворачиваемой одностроковой карточкой** с именем,
аргументами и кратким резюме («134 lines», «5 matches», «exit code: 0»).
Развернуть — нажать chevron. Стандартный паттерн tool-use UI:
имя + результат всегда видны, тело по требованию.

Ещё одно открытие — **слабые модели не любят длинные системные
промты**. Пробовали расширить промт описанием всех трёх инструментов
с примерами — бенч просел с 16/16 до 12/16. Откатили обратно к
минимуму, описывающему только read_file (главный); grep и run_tests
модель находит сама когда нужно. Урок: каждое слово в системном
промте — это шанс что модель потеряется в инструкциях вместо
решения задачи. **Краткость промта важнее полноты документации.**

И последнее — стрим. Изначально мы рендерили модельный ответ в
Textual'овский `Markdown` виджет, обновляя его на каждом токене.
Markdown re-парсит на каждом update; для 1k токенов это очередь из
тысячи обновлений, и UI продолжает «печатать» ещё минуту после того
как модель уже idle. Switch на простой `Static` (мгновенный update)
во время стрима, swap на `Markdown` после finalize — теперь
стриминг идёт со скоростью модели, рендеринг подсветки одноразовый
в конце.

## Глава 4 — формат tool calls

В первой реализации tool-use мы парсили текстовый формат регэкспом:
модель эмитила `<TOOL: read_file>...path...</TOOL>`, мы вытаскивали и
исполняли. Работало, но это **наш самопальный формат** — модель его
не знала, мы его учили через few-shot примеры в системном промте.

OpenAI/Anthropic API давно предлагают **native function calling**:
ты отдаёшь схему функций в JSON Schema, модель в ответе возвращает
структурированные `tool_calls` объекты с `id`, `name`, `arguments`.
LM Studio пробрасывает это к qwen2.5-coder, который умеет с обучения.

Переключились. Бенч сначала просел с 21 до 12 — модель в нативном
режиме иначе форматирует свой ответ (SEARCH/REPLACE блоки могут
прийти с 4-space индентом если стрим начинался после tool-result,
маркеры могут быть с 6 или 8 угловыми скобками вместо 7). Несколько
итераций: расслабили regex `<{5,9} SEARCH`, добавили tolerance к
ведущим пробелам, стрипаем `path:` префикс — вернулись к 23/24.

Интересное **наблюдение**: с native function calling **не нужны
few-shot примеры**. В text-based варианте few-shot был критически
важен — модель должна была научиться формату. С native — формат
прописан в JSON Schema, модель его знает с обучения. Системный
промт сократился, а это плюс к производительности на слабых
моделях, которые теряются в длинных промтах.

Лояльность к формату — отдельная история. В прошлой главе мы решали
**unified diff → SEARCH/REPLACE** (формат, который модель понимает
сильнее → меньше отказов). В этой решали **наш кастомный формат →
стандартный API** (формат, который модель **уже знает** → не нужно
учить). Оба перехода ведут в одну сторону: **минимизируй то, что
модели нужно выучить для общения с инструментом**.

Урок: **смотри что у модели уже есть в обучении**. Custom format
кажется простым в реализации, но требует обучения модели через
few-shot. Native format сложнее в реализации (надо разбираться с
API), зато бесплатно использует то что модель уже знает.

## Глава 5 — модель выбираем не по размеру

После того как v0.2 устаканилась, прогнали бенч-сюиту из 24 тестов
(15 типовых правок + multi-file navigation + 7 поведенческих +
1 xfail на grep) на шести моделях через идентичный harness в LM
Studio. Отличается только модель — архитектура агента, формат
patch'ей, tool calling, контекст 16k — везде то же самое. Железо:
RTX 5060 Ti 16 GB, Xeon E5-2678 v3, Ubuntu 24.04, спек-декодинг
только там где есть совместимая draft-модель.

| модель | размер | arch | quant | спек | pass | время |
|---|---|---|---|---|---|---|
| 🥇 **google/gemma-4-26b-a4b** | 26B MoE (4B active) | gemma4 | Q4_K_M | выкл | **24/24 (100%)** | ~120s |
| 🥈 qwen2.5-coder-14b | 14B dense | qwen2 | Q4_K_M | вкл (qwen2-0.5B draft) | 23/24 (96%) | **~45s** |
| 🥉 openai/gpt-oss-20b | 20B dense | gpt-oss | MXFP4 | выкл | 21/24 (87.5%) | ~106s |
| qwen3.5-9b | 9B dense | qwen35 | Q4_K_M | выкл | 19/24 (79%) | ~73s |
| qwen3.6-35b-a3b | 35B MoE (3B active) | qwen35moe | Q4_K_M | выкл | 13/24 (54%) | ~542s |
| qwen3.5-35b-a3b | 35B MoE (3B active) | qwen35moe | Q4_K_M | выкл | 10/24 (42%) | ~303s |

### Что бросается в глаза

**1. Лучший результат — у general-модели, а не у coder.** Google
gemma-4-26b-a4b, без всякой coder-специализации, без спек-декодинга,
взяла **24/24** — включая тот единственный кейс (`change_return_type`)
который coder-14b стабильно валил. Это **ломает две наших гипотезы
сразу**:
- ~~«coder-tuning обязателен для patch-tasks»~~ — нет, не обязателен
- ~~«MoE на LM Studio не работают»~~ — gemma это MoE 4B active,
  и работает

**2. Coder-14b со спек-декодингом остаётся Pareto-выбором.** 96%
качества, в 2.5× быстрее gemma. Если 4% разрыв качества вы
готовы платить скоростью — gemma. Если важнее latency — coder.

**3. Размер модели всё ещё не главное.** qwen3.5-9b dense general
(79%) обходит обоих своих больших родственников qwen3.5/3.6-a3b
(42-54%). 9B обходит 35B по обоим критериям одновременно — на
нашем стеке.

**4. MoE и MoE — две большие разницы.** Все три MoE-модели в
нашем замере — Q4_K_M, в одном LM Studio, на одном железе:
- gemma-4-26b-a4b (4B active): **24/24, 120s**
- qwen3.6-35b-a3b (3B active): 13/24, 542s
- qwen3.5-35b-a3b (3B active): 10/24, 303s

Между ними ×3 по скорости и ×2.4 по качеству. **Реализация MoE
у Google и Alibaba разная.** GShard/Switch исследования Google
тянутся с 2020, у них есть преимущество в routing-механиках.
Один tokenizer-family (`qwen35moe`) делает хуже чем другой
(`gemma4`). Без файнтюна или внутренних бенчей мы не сможем
изолировать что именно — но факт зафиксирован.

**5. Спек-декодинг даёт компаунд-эффект.** Голая 14B без спек
была бы медленнее 9B (40-50 tok/s vs 52). Но draft в 0.5B даёт
+60-90% к скорости target — итог 80 tok/s, обходим всех dense.
Это урок не про размер модели, а про **подбор пары target+draft**.
Без подходящего draft у qwen3.5-9b или gemma спек не подключить —
производители не выпустили совместимую маленькую draft в этих
сериях. Если бы выпустили — Pareto-фронт сместился бы.

### Почему general-модели проседают на patch-tasks

Три причины, в порядке важности:

1. **General-модели «причёсывают» код при копировании.** Coder-
   модель на запрос «исправь функцию» копирует строку байт-в-байт
   в SEARCH-блок. General-модель часто нормализует: меняет
   кавычки, выравнивает отступы, добавляет пробел перед двоеточием.
   Наш applier допускает whitespace-tolerance, но не quotes —
   SEARCH перестаёт совпадать с source.

2. **MoE прыгает между экспертами по токенам.** Каждый токен
   маршрутизуется через свой набор экспертов. На длинной
   последовательности отступы и маркеры могут поплыть в середине
   ответа. Dense coder на коде монотоннее.

3. **Слабее видела SEARCH/REPLACE на pretraining.** Coder-датасет
   qwen2.5 включает кучу git-merge-конфликтов и diff'ов. General
   видели меньше → хуже воспроизводят синтаксис маркеров.

Доминирующая — №1. Поэтому простые задачи (одно изменение в одной
функции) на 35B-MoE проходят, а сложные (extract helper, рефактор)
— нет: больше токенов копируется дословно → больше шансов на
«нормализацию».

### Куда что подходит

| модель | роль в системе |
|---|---|
| gemma-4-26b-a4b | топ качества, если готов платить 2.5× времени |
| qwen2.5-coder-14b | дефолт: Pareto-выбор по скорости + качеству |
| gpt-oss-20b | кандидат для plan/review/explain в v0.3 dual-model |
| qwen3.5-9b | fallback для low-RAM setup'ов (~5-6 GB Q4) |
| 35B-MoE qwen (3.5/3.6) | пропустить — не наш use case |

### Главный урок

«**Купи модель пожирнее**» — не работает на coding-задачах с
формальным форматом ответа. Что работает (в порядке важности
по нашему бенчу):

1. **Лотерея провайдеров** — gemma-4 vs qwen3.5/3.6 одного класса
   (все MoE, все ~30B-class, все Q4) различаются по качеству в
   2.4 раза. Конкретные команды собирают данные и тюнят routing
   по-разному. Не доверять «MoE как класс хорош/плох» — проверять
   конкретные модели.

2. **Coder-специализация даёт скорость через спек.** Сама по себе
   она НЕ обязательна для качества (gemma-general берёт 100%), но
   она часто идёт в комплекте с маленькими draft-моделями той же
   серии (qwen-coder-0.5B как draft к 14B). Получаешь ×2 к скорости.

3. **Dense — для скорости, MoE — для качества.** На нашем стеке
   все dense быстрее MoE по wall-clock. Но gemma-MoE показала
   что качественный MoE-routing может вытащить generic-модель
   в топ. То есть **архитектура — не приговор**, важна конкретная
   реализация.

4. **Спек-decoding с матчинговым draft** — ×1.5-2 к скорости
   target без потери качества. Самый недооценённый рычаг.

5. **Сжатие Q4_K_M** — sweet spot, не Q8 и не FP16. Q6 даёт
   ~1% качества за ~30% памяти.

Главное: **меряй каждую модель сам**. Бенчмарки от вендоров
показывают пиковое поведение модели на их датасете. Тебе нужно
поведение на **твоей задаче** в **твоём стеке**. Сорок минут
автоматизированного бенча сэкономят недели гаданий.

## Глава 6 — Hallucinations не там где их ищешь

День сюрпризов. Тестовый сценарий: юзер спрашивает «где в проекте
сжимается контекст?». Модель отвечает «в методе `mark_compacted`
класса Session». **Это ложь** — `mark_compacted` только сохраняет
snapshot цифр, реальный compact живёт в `StepAgent.compact()` в
другом файле.

Это **misattribution**, не fabrication. Имя `mark_compacted` —
реальное. Класс — реальный. Файл — реальный. Только описание того
**что метод делает** — выдумано. Никакое количество правил «не
выдумывай имена» этот класс багов не ловит.

Прошли через **четыре** уровня попыток фикса:

1. **Tighter grounding rules** — «before claiming what a method
   does, call read_file». Модель проигнорировала.
2. **Concrete anti-examples в промте** — добавили «if MAP shows
   mark_compacted, don't answer with compact». Помогло для прямых
   запросов, не помогло для retrieval («где сжимается?»).
3. **Per-mode temperature 0.1** для ask. Помогло не выдумывать
   откровенную чушь, но не отучило от misattribution.
4. **Финал — обогащение MAP**. Добавили первое предложение
   docstring как комментарий к каждому символу. Теперь модель в map
   видит:
       async def compact(...) -> str | None  # Summarize history into a short note and replace it.
       def mark_compacted(...) -> None  # Anchor the footer budget to the post-compact state.
   и **сама** выбирает правильный — без read_file, без правил.
   На 15/15 формулировок за 3 прогона.

Урок: **недостаток информации в map нельзя лечить правилами в
промте**. Модель знает что видит. Если она не видит семантику —
никакая нотация в инструкциях её не научит выбирать правильный
символ. Обогащай источник данных, а не уговаривай модель читать
лучше.

Параллельно нашли структурную проблему: модель в ask-режиме
утверждала «classifier.py классифицирует задачи во flow» и писала
**фейковый код** `task_type = classify(task)` в `StepAgent.ask()`.
classifier — standalone утилита, нигде не вызывается. Map показывал
обе сигнатуры (classify + ask), но связь между ними была невидима.

Решение: добавили в map `imports: ...` строку для каждого файла,
показывающую intra-project зависимости. После этого модель видит:
`agent.py imports: code_scalpel.config..., code_scalpel.llm.adapter...`
— **никакого classifier нет в списке**. И в промте правило: эта
строка — ground truth, не клейми «X uses Y» если Y нет в imports.

Этот фикс закрывает целый класс «угадай вызывается ли». Generic,
работает на любом Python-проекте, дёшев в плане токенов.

**Главный takeaway**: weak local LLM работает не хуже сильной если
**ей дать видимость**. Промт — это **протокол поведения**. MAP —
это **знание о мире**. Не путать их.

## Глава 6.5 — Память: почему пишем своё (и почему не сейчас)

Памяти агенту сейчас нет — `_history` живёт в RAM до /new или
выхода из сессии. На v0.3 запланирован memory layer: persistent
recall между сессиями, дедуплицируемое хранение фактов о проекте,
retrieval по релевантности.

Сначала мы хотели взять `mem0ai` — узкая библиотека на 150 строк
кода, API из двух методов (`add` / `search`), под капотом Chroma +
LiteLLM. Outsourced 30% work. Запустили **spike в /tmp** на полдня.

Результат spike: **mem0 SKIP**. Четыре причины:

1. **API instability**. В версии 2.0.2 они сломали свою же
   документацию: `search(user_id=...)` стал `search(filters={...})`.
   Доверия к стабильности нет.
2. **LM Studio несовместимость**. mem0 хардкодит
   `response_format={"type": "json_object"}` в запросе для извлечения
   фактов. LM Studio такое не поддерживает — требует `"text"`. Чтобы
   запустить — нужен monkey-patch, который не переживёт следующий
   `pip upgrade`.
3. **Дедупликация не работает**. Главная причина брать mem0 был
   автоматический dedupe противоречий («раньше юзал qwen» → потом
   «перешёл на gemma» → старая запись должна обновиться, не
   дублироваться). В 2.0.2 vector-путь просто кладёт обе записи
   рядом. Advertised feature отсутствует в реализации.
4. **138 MB зависимостей**. numpy, grpcio, sqlalchemy, qdrant-client,
   posthog (трекинг по умолчанию!). За что? За API из двух методов
   которые мы можем написать сами за день.

Вывод: пишем `code_scalpel/memory.py` на ~150 LOC. sqlite для fact
rows (persist), Chroma или Qdrant через abstract interface (не
lock-in), наш собственный `LLMAdapter` для contradiction-check
prompts. Время реализации: половина дня. **Без 138 MB legacy и без
зависимости от чужого release schedule**.

**Почему не делаем прямо сейчас**: 297 unit-тестов и 31/31 LLM-бенч
зелёные БЕЗ memory layer'а. Юзер не упирался в «модель забывает что
вчера говорил» — это не блокер. Memory нужен как **фундамент для
autonomous-mode** (run-plan loop, который сам помнит progress) — а
до autonomous'а ещё несколько других кусков. Берём memory.py когда
он реально станет нужен, а не «потому что в plan'е написано».

**Lesson**: spike перед интеграцией экономит недели. Без spike'а
mem0 пришёл бы как dependency, мы бы две недели возились с monkey-
patch'ами и узнали бы про сломанную дедупликацию посередине пути.
Полдня в /tmp — лучшая инвестиция последней недели.

## Глава 7 — Параллельные агенты как методология

Этот день начался с того, что юзер сказал: «что стесняешься, агентов
побольше юзать». До этого мы запускали по одному агенту на задачу.
После — стандартный приём: на каждой итерации сразу 3-5 агентов
параллельно, каждый со своей зоной.

Один день — 7 агентов в параллель:
- **Iterative patch loop в agent.py** (implementation)
- **PlanCard widget в TUI** (implementation)
- **Slash-команды /tasks /system** (implementation)
- **Tab/focus navigation fix** (research + fix)
- **mem0 spike в /tmp** (research only, isolated)
- **AST tree-sitter spike в /tmp** (research only)
- **Coverage gap analysis** (read-only audit)

Результат: 5 коммитов в main за один проход, 13 новых тестов, 0
конфликтов, 0 регрессий бенча.

**Что сработало:**

1. **Файловые границы**. Каждому агенту явно сказано: трогать ТОЛЬКО
   эти файлы, НЕ ТРОГАТЬ ВОТ ЭТИ. Например: implementation-агент
   patch loop'а работал только в `agent.py`/`config.py`/`test_agent.py`.
   TUI-агент Tab — только в `tui/widgets/output.py`, `tool_use.py`,
   `test_app.py`. Параллельный slash-агент — только `tui/app.py` +
   `test_app.py`. Конфликт `test_app.py` между slash и Tab —
   sequential commits на main, без merge headaches.
2. **Research отдельно**. mem0/AST спайки целиком в `/tmp`, ничего в
   main. Отчёт текстом, решение принимает основной агент.
3. **Brief как для нового сотрудника**. Каждый агент-инструкция
   начинается с контекста («это TUI на Textual, есть такие-то
   компоненты, посмотри файл X»), потом задача, потом success
   criteria и анти-критерии («НЕ ТРОГАЙ эти файлы»). Без брифинга —
   агент пишет в стиле «как я понял», с брифингом — попадает в
   проектный стек с первой попытки.
4. **Background mode**. Все агенты запускаются с
   `run_in_background=true`. Основной агент работает дальше, агенты
   присылают completion notification. Никаких sleep'ов, никакого
   polling.

**Что неожиданно вылезло:**

- **mem0 spike сэкономил недели**. Без спайка мы взяли бы mem0 как
  dependency в v0.3 (роадмап так и говорил). Спайк показал что
  v2.0.2 сломала свои же доки, hardcoded `response_format` ломает
  LM Studio, дедупликация вообще не работает — главная причина брать
  библиотеку отпала. Решение: пишем свой `memory.py` (~150 LOC).
  Без спайка — обнаружили бы это через две недели интеграции.
- **Coverage-агент нашёл что мой own код 81%-covered**, с конкретным
  списком пяти must-have тестов. Half a day of solo audit work, done
  in 90 sec by an agent.

**Лимит**: агенты ломаются на задачах требующих UI-feedback
(«посмотри как выглядит и улучши») — там probe + spy лучше делать
самому. И на задачах с тонкой настройкой промта — там итерации
быстрее в живой сессии.

**Главный takeaway**: распараллеливание агентов — это не «дать им
писать код за тебя». Это **разделение труда**: один думает, другой
проверяет, третий исследует, четвёртый смотрит coverage.
Все одновременно. Скорость роста проекта вышла раз в 5× быстрее.

## Глава 8 — Карта как редактор, а не как энциклопедия

Сессия началась с того, что probe-скрипт прогнал 8 живых сценариев на
LM Studio и **все восемь** упали с одной и той же ошибкой:
`n_keep >= n_ctx`. Полный symbol map проекта весил ~14k токенов и
ехал в каждом turn'е — окно 16k забивалось до того как модель успевала
сказать первое слово.

Очевидное решение — урезать map — упирается в то что урезать-то и
нечего: docstrings, imports, сигнатуры — всё это мы сами же
обогащали в главе 6 чтобы модель не выдумывала. Каждая строка
заработана потом.

Сдвинули угол: **не урезать данные, сменить access pattern**. Слабая
LLM — не Memorizer, который держит проект в голове. Это пользователь
с Ctrl+F. В каждом turn'е едет **overview** — пути и количество
строк, ~375 токенов на 50 файлов. Когда модели нужны детали — она
вызывает новый инструмент `map_file(path)`, и получает тот самый
толстый блок «сигнатуры + docstrings + imports» только по одному
файлу. Информация та же. Поток — иной.

После этого приехали `goto_definition` и `find_references` — два
инструмента, скопированные прямо с IDE: «знаю имя, веди меня к
определению» и «где это вообще упоминается». Find_references
намеренно текстовый, не AST-only — когда человек спрашивает «где
упоминается X», он ждёт что TODO-комментарий и import-строку
посчитают, а AST их выкинет.

Урок: **дай модели инструменты, которыми пользуется живой
разработчик, а не инструменты которые удобно завернуть в API**.
Overview → map_file → read_file → goto_definition → find_references →
grep — это последовательность, которой человек ходит по проекту в
любой IDE последние 20 лет. Слабая модель в эту последовательность
встаёт сама, без объяснений в промте.

## Глава 9 — «Кто ты» и идентичность через примеры

День после navigation refactor'а юзер набрал в TUI простое «кто ты».
Модель отвечала: **«Ты — ассистент, разработанный для работы с
кодом…»**. Три бага за одну реплику: грамматическое лицо не то
(модель про себя в третьем), название проекта потеряно, и сам ответ —
дословный перевод английской системной шапки.

Сначала возник соблазн дописать в промт правило: «когда тебя
спрашивают кто ты — отвечай Я — code-scalpel». Но это не одно
правило, это класс багов. Промт был полон рядовых упоминаний
«адресуй пользователя на ты». Модель этот «ты» в свой лоб и
прицепила — слабые LLM не различают «ты к собеседнику» от «ты как
self-label».

Финал: добавили в системный промт явный Identity-блок с шаблонами
**в первом лице** на обоих языках («Я — code-scalpel…», «I'm
code-scalpel…») и одной строкой объяснили — правило «ты/вы» про
**обращение** к пользователю, не про **самоописание**. Probe-сценарий
`identity` теперь ловит регрессию каждый прогон.

Урок: **слабая модель учится по примерам, а не по правилам**.
Можно написать «не переводись дословно из английской шапки» — модель
найдёт способ перевестись. Можно показать ей готовую фразу в нужной
форме — она её скопирует. Examples-driven prompting на 14B бьёт
rule-driven с разгромом.

## Глава 10 — ↑/↓ это история, а не дропдаун

textual-autocomplete по умолчанию ловит ↑ и ↓ как «двигайся по
выпадающему списку команд». Логика разумная для веб-формы. Для
шеллового prompt — катастрофа. Юзер жмёт ↑ ожидая предыдущую команду
из истории, и получает скачок внутрь slash-меню.

Подсмотрели как с этим живёт bash. Там ↑/↓ — это **command history**,
а completion вызывается явным Tab. Никакой неоднозначности. Сделали
`HistoryInput` — subclass Input с priority-bound ↑/↓: dropdown их
больше не видит. ↑ откатывает на предыдущую отправленную команду,
↓ ведёт вперёд по истории, ↓ за самой свежей — возвращает черновик,
который юзер набирал до того как нажал ↑. HISTCONTROL=ignoredups
для подряд идущих дубликатов — bash-style, как и должно быть.

Урок: **правильная interaction model для terminal-shaped UI — та,
которой bash уже научил пользователя**. Виджеты из веб-мира приносят
свои биндинги, и эти биндинги часто конфликтуют с двадцатилетней
мышечной памятью. Когда что-то в TUI «не так», спроси сначала — а
как это работает в шелле.

## Глава 11 — FTS5 и тихая дефолтная конъюнкция

Memory-слой в той же сессии получил wiring в TUI: `/remember`,
`/recall`, и auto-recall — на каждом turn'е StepAgent ищет в
SQLite/FTS5 заметки релевантные текущей задаче и подмешивает их в
сообщение.

Первый прогон auto-recall'а: ничего не находится. Никогда. Юзер
сохраняет заметку «ruff format before commit», потом спрашивает
«what to do before commit?» — пусто. Хотя слова `before` и `commit`
явно общие.

Полез смотреть FTS5. **По умолчанию multi-token query — это AND**:
`before commit do what` означает «найди документы где есть ВСЕ четыре
слова». Документация это, конечно, говорит — но мелким шрифтом и не
там где первый запрос. Все примеры в туториалах однословные,
поэтому ты этого не замечаешь пока сам не наступишь.

Лечится одной строкой: токенизируем задачу по `\w+`, переписываем как
`"tok1" OR "tok2" OR ...`. Один общий значимый токен — и заметка
всплывает. Релевантность ранжирует bm25 как обычно, AND-конъюнкция
просто убрана.

Урок: **узнаешь библиотеку не из README, а из первого реального
вызова**. mem0 в spike'е (глава 6.5) показал то же самое — advertised
feature отсутствовал в реализации. FTS5 — наоборот, фича присутствует
ровно как описана, но default настроен под другой класс задач (поиск
в документах), не под наш (recall по короткому запросу). Полчаса
real usage даёт больше чем день чтения docs.

## Глава 12 — Калибровочный drift: бенч помечает то, что улучшилось, как регрессию

Прогнали v0.3 LLM bench на свежей системе с navigation-нав'ом
(overview + map_file drilldown) и memory wiring. Цифры — 25/31
проходит. На v0.2 было 30/31. Снаружи это выглядит как регрессия
на 16%, но это не она. Это **калибровочный drift**.

Разбираем `test_qwen_admits_missing_method`. Сценарий: модель
получает класс `Cache` с методами `get` и `set`, юзер спрашивает
«где реализован метод `.summarize()`». Тест ожидает фразу вроде
«не нашёл / отсутствует / нет такого». На v0.2 модель отвечала
коротко «такого метода нет» — тест зелёный. На v0.3, после того
как мы дали ей `map_file` и явные grounding rules, она отвечает:

> Похоже, метод `.summarize()` у класса `Cache` действительно не
> реализован в проекте. Если вам нужно добавить этот метод, вы
> можете сделать это следующим образом: …

И дальше — SEARCH/REPLACE-пример как добавить. Модель **сказала
правильное** («не реализован») И **переусердствовала**, предложив
имплементацию. Старый assertion `_admits_missing` метит fail,
потому что ищет «не нашёл» как короткий ответ, а тут — ответ с
pretext'ом, готовый к копи-пасту. Это не galluсination и не
регрессия качества. Это другая шапка ответа.

Похожая история у `picks_correct_method_among_similar_names`,
`history_three_turn_topic_continuity` и трёх patch-задач
(`rename_function`, `add_missing_import`, `remove_unused_import`).
В patch-задачах причина другая: на v0.2 модель сразу видела полный
символьный map в каждом turn'е и рисовала SEARCH прямо оттуда. На
v0.3 она получает overview, потом обязана сделать
`map_file → read_file → SEARCH`. Иногда промежуточные tool calls
накапливают ошибку — SEARCH-блок не сматчился побайтно, потому что
модель скопировала из map_file-вывода (signatures) вместо
read_file-вывода (тело). Это тренируемое поведение, но
bench-fixture ожидает прохождение с первого выстрела.

**Урок**: при крупной перестройке prompt/context structure
автоматический бенч — это не критерий. Он показывает где ассерты
устарели вместе с моделью, и где **реальная** регрессия началась.
Разделить эти два класса без человека нельзя. Cherry-picking
руками первых трёх failing replies: две — «улучшилось, но мимо
assertion», одна — реальное падение точности SEARCH'а. Без этого
просмотра у нас была бы тревожная цифра «упали 6 тестов» и
неправильное направление правки.

В план записано: v0.3 LLM bench нуждается в **recalibration** под
новые формы ответа (over-helpful, navigation-driven SEARCH).
Сравнение с v0.2 невалидно — там был другой кодовый путь. Честное
сравнение требует синтетического бенча с переключаемым режимом
context'а: full-map vs overview, один и тот же набор задач, одна
и та же модель. Это работа на отдельный заход.

## Глава 13 — Терминал ≠ web view: путь к собственному рендеру Mermaid

Юзер увидел один из ответов модели — пять экранов ASCII-art с
файловой структурой проекта вместо нормальной диаграммы потоков.
Сказал: «давай нормальный Mermaid». Логично — модель умеет писать
Mermaid, юзер привык видеть его в VS Code / Obsidian / GitHub.
Кажется, что задача техническая: поймать ```mermaid``` блок,
прогнать через рендер, показать в чате.

Первая попытка — поставить `@mermaid-js/mermaid-cli` (`mmdc`),
который компилирует Mermaid в PNG через headless Chromium. Из
PNG в терминал перевести можно через `rich-pixels` —
Unicode half-blocks. Работает, но требует Node.js + полтора
гигабайта на Chromium. Compliance-сегмент сразу отваливается.

Второй заход — облачный fallback на mermaid.ink. Это HTTP-сервис
который рендерит и отдаёт PNG. Не нужен Node, не нужен Chromium.
Но это сетевая зависимость в сердце local agent'а. Юзер сразу:
«не-не, не хочу наружу».

Третья развилка прозвучала естественно: «VS Code же умеет, и
Obsidian умеет, может через них?». Здесь важный момент. VS Code и
Obsidian умеют **потому что они Electron**. Внутри — браузер,
внутри браузера — `mermaid.js`. JavaScript-движок, который парсит
Mermaid и рисует SVG. Терминалу повторить нельзя — он не запускает
JS, не рендерит SVG. Все «терминальные Mermaid рендеры» которые
ходят по интернету в итоге либо шелятся в `mmdc`, либо
проксируют в облако.

Четвёртый и принятый вариант — **написать свой**. Не полный
Mermaid (грамматика большая, плюс layout алгоритмы), а
**подмножество** из трёх типов которые покрывают реальные
сценарии code-агента:

- `flowchart TD/LR` — связи, workflow, dependency graph. Парсим
  `A[Label] --> B[Label]`, ромбы для решений, метки на стрелках.
  Layout — BFS rank → колонки или строки → ASCII боксы и линии.
- `sequenceDiagram` — путь пользователя, request/response между
  сервисами. Actors как колонки, время сверху вниз, стрелки
  между. Совсем другой layout, тот же парсер недостаточен.
- `classDiagram` — структура классов, иерархия наследования,
  composition/aggregation. Боксы с тремя секциями (имя / поля /
  методы), стрелки с разными стилями (`<|--`, `*--`, `o--`).
  Layout близок к flowchart (rank-based), но shape бокса другой.

Остальное (stateDiagram, gantt, journey, gitgraph, mindmap,
erDiagram) — **out of scope**. Модели в system prompt'е прямо
запрещаем эти типы. Для состояний — flowchart с decisions.
Это компромисс: терминальный ASCII не позволяет тонкости которые
GitHub Mermaid-renderer показывает в HTML. Лучше иметь три
работающих типа, чем семь сломанных.

Первый заход в плане — flowchart как самый частый. classDiagram и
sequenceDiagram расширяют сразу за ним. Все три используют общий
парсер инфраструктуры и общий layout engine; различия по форме
бокса и направлению стрелок изолированы в отдельных модулях. Это
ровно тот «плагин-friendly» design что и в JobRegistry / Skills /
Memory — ядро ничего не знает про конкретный тип диаграммы.

**Урок**: «X тоже умеет, давай через X» — это часто запрос
заглянуть под капот X. Если X — это GUI с JS-движком, а мы — TUI,
то «через X» невозможно по определению. Узкий собственный рендер
под наш use case оказывается короче и предсказуемее любой
проксии. Параллель с mem0 (глава 6.5) — тогда мы тоже хотели
готовое и в итоге написали свой `memory.py` ~166 LOC. Здесь то же
паттерн: ограниченная фича, контролируемая нами, работает
оффлайн без внешних зависимостей.

## Глава 14 — Когда промпт-правило не помогло: post-hoc guard

В главе 6 мы лечили hallucinations добавляя данные в карту
проекта (docstrings, imports). Это сработало для большинства
случаев — модель видит реальный signature и docstring, не
выдумывает методов. Но один сценарий не закрылся даже после
четырёх итераций prompt-правил: **vague show-code**.

Юзер пишет «покажи как работает `summary_line()`». В map есть
сигнатура `summary_line(self) -> str` и docstring. Модель видит
их и… не вызывает `read_file`. Вместо этого пишет «правдоподобный»
код метода — судит по имени + docstring какой должна быть
реализация. Иногда угадывает, чаще нет.

Промпт уже содержал жёсткие grounding rules: «Pattern recognition
is NOT a source of truth», «call read_file every single time you
reproduce more than a signature». Не помогло. xfail-тест
`test_qwen_reads_file_even_for_vague_show_code` так и висел.

Реальный фикс — **post-hoc guard в коде агента**, а не очередная
прошивка в промпт. Логика простая: после получения финального
ответа модели смотрим — есть ли в нём fenced python или
SEARCH/REPLACE блок, нацеленный на файл из проекта. Если есть —
проверяем history tool-calls: вызывался ли `read_file` для этого
файла в текущем или прошлых turn'ах. Если нет — отклоняем ответ
и шлём модели re-prompt: «прочитай файл сначала, потом
перепиши». Cap — 1 retry, упрямую модель в цикл не загоняем.

Симптомы исчезли. Тест `test_qwen_reads_file_even_for_vague_show_code`
стал стабильно зелёным. Открыт через `agent.enforce_read_before_show`
по умолчанию.

**Урок**: prompt-инструкции работают как **soft constraint**.
Когда они стабильно не выполняются — добавлять ещё правила
бесполезно (мы это поняли только на 4-й итерации, потеряв в
качестве промпта читаемость). **Hard constraint** в коде агента —
дёшево, измеримо, не зависит от прихоти модели. Но HOOK имеет
ограниченную область: он ловит fenced code/SEARCH без read_file.
Если модель фабрикует pure-prose-ответ («uses.py contains no
imports» без блока кода), HOOK слеп — там надо другой механизм.
Бенч `remove_unused_import` остался xfail именно по этой
причине. Каждый класс ошибок требует своего guard'а.

## Глава 16 — Example trap: слабая модель копирует то, что мы привели как «хорошее»

Probe-сценарии 2026-05-12: после переделки user_message (только
task, без auto-инжекта файлового листинга) и ужесточения
identity-блока в промпт получили **3/10**. Раньше было 7/10.
Модель буквально на каждый запрос отвечала «Сейчас посмотрю...»
и больше ничего.

Это была не случайная регрессия. В промпте мы добавили:

> ABSOLUTE BAN on task replies starting with "Я — code-scalpel".
> For tasks, start with the action ("Сейчас посмотрю...",
> "Looking at...", or just the result after tools come back).

Видишь подвох? Мы дали модели **конкретный пример** того, как
надо открывать ответ. Дальше она его **скопировала literal** и на
этом остановилась. Кавычки вокруг примера не помогли — модель
читает их как «вот шаблон, используй».

Это **тот же elephant trap**, что и в главе 9 с identity. Тогда
мы перечислили в промпте «вот эти запросы — НЕ identity» — модель
залипла на них и начала отвечать «Не понял, переспроси?» на
легитимные context-вопросы. Урок мы тогда вынесли, но
обобщить — забыли.

Финальный фикс — убрать любые конкретные opening-фразы:

> ABSOLUTE BAN on task replies starting with "Я — code-scalpel".
> For tasks, call the appropriate tool first; the result of the
> tool call IS your way to start working. Don't announce what
> you're about to do; just call the tool.

После переделки 7/10. Главная регрессия ушла. Уровень
performance вернулся.

**Правило для weak-LLM промптинга**: НИКОГДА не давай конкретный
positive example, который модель может скопировать буквально.
Только **rule** + **anti-example в виде ban'а**. Если хочется
показать «как надо» — описывай словами («open with an action
verb, not with a self-reference»), не цитатой. Слабая модель
treats цитаты как императив.

Negative examples тоже опасны (глава 9), но они хотя бы
self-limiting — модель не повторит то, что было названо
запрещённым. Positive examples — самое мерзкое: модель
буквально повторяет их as-is.

В bench утащил два anti-теста под эту регрессию:
`test_qwen_task_does_not_open_with_identity_prefix` и
`test_qwen_doesnt_apologise_instead_of_calling_tools`. Probe
остаётся как «видеть глазами» инструмент; bench пинит конкретные
failure shapes так что в CI они не повторятся незаметно.

## Глава 17 — Два пути одного режима: probe и TUI давали разный ответ

2026-05-12, после регрессии из главы 16, появилась более тонкая
проблема: probe ставил 7/10 на типовых сценариях, юзер в TUI
показывал скриншоты «всё работает». Я начал делать вывод «модель
отказывается вызывать project_map» по probe-логам — юзер
возразил: «не верю, у меня всё нормально». Скриншоты подтвердили
— в TUI модель действительно вызывала тулзы как надо.

Разгадка лежала на расстоянии grep'а. У `StepAgent` было **два
независимых tool-loop'а**: `ask()` через `_chat_loop_with_hook` →
`_chat_loop` (non-stream, плюс HOOK enforce-read-before-show) и
`stream_ask()` — отдельная реализация для TUI, без HOOK'а вообще.
Probe гонял `ask()`. TUI стримил через `stream_ask()`. Два кода —
два разных поведения. Защита, которую мы добавили против
hallucinated-патчей (HOOK), работала только в probe и тестах.
Живой юзер в TUI был не защищён.

Это классическая «duplicated execution path» проблема — она
встречается везде где есть streaming/non-streaming пара. Кажется
что код «почти одинаковый», и сначала так и есть. Потом одну
ветку трогают (HOOK, compression, plan-save, usage tracking),
другую забывают. Через несколько таких правок поведения
расходятся достаточно чтобы probe и реальное использование жили
в разных вселенных.

Унификация: `stream_ask` стал canonical engine, `ask` — тонкая
обёртка над ним, собирающая stream в `StepResult`. Tool-loop
вынесен в `_run_tool_loop` чтобы HOOK мог откатить турн и
повторить его без копирования управления round'ами. HOOK теперь
работает во всех режимах (ask/plan/code/review/code-retry) через
один путь. Когда он срабатывает посреди стрима, агент yield'ит
`RetryNotice(path)` — новое typed event — а TUI рисует inline
`↻ Re-reading <path> before patching` чтобы юзер не видел
«дублирующийся ответ» без объяснений.

**Правило**: streaming и non-streaming не должны быть двумя
независимыми реализациями. Один движок отдаёт events; обёртка
для non-streaming просто аккумулирует их в финальный объект.
Иначе любая защита/инструментация будет периодически
«пропадать» в одном из путей, и узнавать об этом будем по
расходящимся probe vs prod метрикам.

## Глава 18 — read_file как окно, а не как файл

Долго read_file делал одно: брал первые 400 строк и возвращал
их с line-numbered префиксом. Для маленьких файлов работало;
для больших обрезал по строке 400 с надписью «… N more lines».
Юзер задал правильный вопрос: «а если в файле 10 000 строк? я бы
ожидал что read_file даст возможность считать кусок — конкретные
строки либо область вокруг имени».

Ровно ту же эргономику дают `awk 'NR==100,NR==150'`, `sed -n`,
`grep -A 20 -B 20`. Это давно понятые удобства для человека за
терминалом. Слабой модели они не менее нужны — у неё всё то же
ограничение context budget, что и у нас, только ещё жёстче.

Финальная форма — три режима в одной функции, выбираются по
тому, какие аргументы переданы:

1. **whole** — `read_file(path)`. До `max_lines` сверху,
   труncation footer.
2. **window** — `read_file(path, start_line=10, end_line=12)`
   или одна из границ; вторая по умолчанию тянется до края или
   `max_lines` от start'а. Используется когда у нас есть hint
   (упавший тест с `path:N`, project_map выдал `class X` на
   строке 42 — берём window вокруг).
3. **find** — `read_file(path, find="target", context=20)`.
   Возвращает все вхождения substring'а с контекстом до/после
   каждого, причём overlapping spans объединяются. Это
   эквивалент `grep -n -B 20 -A 20`. Используется когда модель
   знает имя но не знает где оно лежит.

Output всегда с 1-based line numbers — чтобы цитата `path:N` и
SEARCH-блок патча ссылались на реальные строки.

Эта правка важна не сама по себе, а как пример другого
принципа: **прежде чем чинить промт, обогащай данные**.
Hallucinations часто бывают от того что у модели нет
дешёвого способа получить нужный кусок. Заставлять её
вытаскивать функцию через `read_file(whole)` который читает
500 строк и обрезает на 400 — это просить fabricate.
Дай ей `find=` — она перестанет фантазировать, потому что
правильный вариант стал дешевле и удобнее.

## Глава 19 — Token counting: heuristic vs реальный usage

Юзер прислал скриншот: `↑3k ↓0k (3k total) · 11m 22s`. Сессия
11 минут, тулзы вызывались, ответы приходили — а completion
tokens записаны как ноль. Cause-of-the-bug отыскался быстро.

`stream_ask` в TUI считал completion_tokens как
`len(full) // 4`, где `full` — текст, накопленный из
`TextDelta` чанков последнего round'а. Но финальный round мог
заканчиваться tool_call'ом без всякого финального текста.
Юзеру тулзы выполнялись, патчи накладывались — текста не было,
heuristic давал ноль.

Char-counting вообще ненадёжен — он считает символы, не токены,
и пропускает всё что не текст: tool_call arguments, function
names, system overhead. Реальный путь — попросить у провайдера
честные числа. OpenAI-compatible API поддерживает
`stream_options={"include_usage": True}` — финальный чанк
стрима тогда несёт `usage` с `prompt_tokens` и
`completion_tokens`. LM Studio это умеет.

Поток данных: адаптер пробрасывает `usage` через новый
`StreamChunk.usage`. Агент копит цифры через все round'ы
(prompt_total = последний; completion_total = сумма) и в
конце yield'ит `UsageReport(prompt, completion)` как
самостоятельный StreamItem. TUI принимает его и записывает в
`Session.record()` вместо heuristic'а. Fallback на
`len(full) // 4` остаётся для старых провайдеров, но в норме
никогда не активен.

**Правило**: когда есть способ спросить у источника точное
число — не считай его сам. Heuristic — это технический долг
который однажды соврёт молча; и узнаешь ты об этом из
скриншота юзера, а не из логов.

## Глава 20 — navigate vs summarize: два разных ответа на «читай интернет»

Задача поставлена просто: дать агенту возможность читать веб. На практике
«читать» означает разное в зависимости от цели. Первый вариант — скачать
статью, прогнать через summarization-промт и вернуть дайджест. Это хорошо
работает на вопросе «стоит ли использовать библиотеку X» — агенту нужен
вывод, не содержимое. Второй вариант — сохранить статью на диск и дать
модели навигационный доступ: сначала список заголовков, потом конкретный
раздел по имени. Это хорошо работает на технической документации, где
модели нужна точная секция («параметры инициализации», «примеры ошибок»).

Попытка выбрать один режим — ошибка. Если всегда summarize: справочная
документация сжимается до нечитаемого блёрба, конкретные параметры теряются.
Если всегда navigate: ответ на «что лучше — lib A или lib B» требует двух
скачиваний и четырёх вызовов инструмента прежде чем модель скажет «A лучше».

Решение: `fetch_page(url, mode="navigate"|"summarize")` — один инструмент,
два поведения. Дефолт navigate потому что технические статьи чаще нужны
по частям, а summarize дороже (лишний LLM round-trip). Модель передаёт
mode явно когда хочет другой режим — это уже обработанный паттерн в
нашем стеке (аналог `read_file` с тремя режимами из главы 18).

Наблюдение про «анти-банинг»: дело не только в UA-ротации. Главный триггер
бана поисковика — отсутствие задержки между запросами. Живой браузер не
делает 50 запросов в секунду. `WebClient` с jitter'ом и exponential backoff
на 429/503 — это не параноя, а базовая вежливость к сервису. DuckDuckGo
HTML endpoint терпелив, но не бесконечно.

## Глава 21 — write_file как канонический tool вместо SEARCH/REPLACE

Изначально модель меняла файлы через SEARCH/REPLACE блоки в текстовом
ответе: имя файла, fenced block, `<<<<<<< SEARCH` / `=======` /
`>>>>>>> REPLACE`. Парсер вытаскивает блоки, applier применяет. На сильных
моделях работает: блок матчится посимвольно, замена точная. На qwen-14b
ломается на пробелах и табах — модель «знает» как выглядит код, но
воспроизводит с дрейфом, applier отказывается. После N подходов план
рушится.

Перепридумали как `write_file` с тремя режимами:
- `write_file(path, content)` — overwrite/create целиком (для новых
  файлов и небольших правок);
- `write_file(path, content, start_line, end_line)` — заменить диапазон
  строк (хирургия в большом файле без перепечатывания всего файла);
- `write_file(path, content, insert_after_line)` — вставка после строки
  N (0 = prepend).

Это не идея «убрать SEARCH/REPLACE потому что плохо», а наблюдение:
function calling + явные параметры — естественный язык для слабой
модели, а текстовые маркерные блоки — нет. Каждый параметр валидируется
схемой, ошибка лезет как `error: start_line > file length`, модель
учится за один retry. SEARCH/REPLACE же даёт «patch did not apply
cleanly» — модель не знает, где промахнулась.

SEARCH/REPLACE инфраструктура осталась (parser, applier) как legacy
fallback, но из промптов выпилена. mode_code.md теперь учит **только**
write_file. Чек-лист задачи: «ваша задача обязана закончиться хотя бы
одним успешным write_file» — иначе run_plan статус failed.

Побочный эффект: каждый write_file — отдельный tool card в чате с
аргументами. Пользователь видит ровно ту запись которую делает агент:
путь, режим (overwrite/range/insert), длина контента. Раньше блок
SEARCH/REPLACE прятался внутри турна, всплывал только когда модель
эмитила его в тексте; теперь действие видно сразу.

## Глава 22 — Промпты как .md, а не как py-константы

`agent.py` накопил 250+ строк triple-quoted констант: `_SYSTEM_PROMPT`,
`_CODE_MODE_ADDENDUM`, `_PLAN_MODE_ADDENDUM`, `_REVIEW_MODE_ADDENDUM`,
плюс полдюжины `_APPLY_FAILED_PROMPT` / `_TESTS_FAILED_PROMPT` и т.п.
для retry-цикла. Редактировать неудобно: длинная строка пляшет под
auto-format ruff, escape-последовательности в `{paths}` плейсхолдерах
ловят опечатки, ревьюверы пролистывают «там же только текст».

Вынесли в `code_scalpel/prompts/`:
- `system.md`, `mode_code.md`, `mode_plan.md`, `mode_review.md` —
  четыре главных промпта;
- `retry/*.md` — apply_failed, tests_failed, missing_files,
  needs_tests, read_before_show, force_answer;
- `annotate_skills.md` — отдельный промпт для skill-annotation pass.

Модуль `prompts/__init__.py` грузит всё на импорте через
`importlib.resources` и держит как константы. Файлы маленькие (<2KB
каждый), кешировать lazily смысла нет.

Уровень удобства правки: чтобы изменить инструкцию модели по созданию
файлов — открываешь `mode_code.md` в редакторе как обычный markdown,
видишь подсветку, никаких `\n` или экранированных кавычек. Diff в
PR ревью читается без расшифровки. Каждый промпт можно отдельно
скопировать в playground провайдера и тестить руками.

Цена: один уровень indirection (`_prompts.MODE_CODE` вместо
`_CODE_MODE_ADDENDUM`). Тесты пришлось чуть подкрутить — некоторые
ассерты искали «SEARCH» в системном промпте, после рефакторинга
строка переехала в `mode_code.md` и появилась только в коде-режиме.
Это нормальный шум миграции, не аргумент против переноса.

## Глава 23 — Sandbox через bwrap: один subprocess от user'ского /home

Скептик и оптимист в `policy.py` решают вопрос «надо ли спрашивать
пользователя перед shell_exec». Но даже разрешённая команда исполняется
от того же uid, что и сам скальпель — то есть может прочитать `~/.ssh/`,
`~/.aws/credentials`, переписать `~/.bashrc`. Модель не злонамеренна,
но галлюцинации + autonomous-loop = реальный риск.

Решение — bubblewrap (`bwrap`), SUID-less Linux sandbox из пакета
`bubblewrap` на Debian/Ubuntu/Fedora. Один `subprocess.exec` с
обёрткой:

- `--ro-bind /usr /usr`, `/lib`, `/bin`, `/etc` — system tools работают
  (`python3`, `pytest`, `git`), но писать в них нельзя;
- `--bind <project> <project>` — проект rw, тот же путь что и снаружи
  (абсолютные пути не ломаются);
- `--tmpfs /home`, `--tmpfs /tmp` — `~/.ssh` и временные файлы хоста
  не видны вообще;
- `--share-net` — pip / локальный LLM требуют сеть, не изолируем;
- `--unshare-user --unshare-pid --unshare-uts --unshare-ipc` —
  namespace-изоляция остального.

В конфиге `AgentConfig.sandbox = "auto"`: используем bwrap если есть,
иначе работаем без него (на macOS / WSL fallback — нет sandbox-а).
«on» — требовать bwrap (отказать если не установлен). «off» —
выключить независимо от наличия.

Параллельно укрепили `policy.py` pattern-блоками: `cd /...`, `cd ..`,
`>/etc/...`, `cp X /usr/...` отрубаются даже когда sandbox выключен.
Дешёвая «вторая линия» для систем где bwrap нет — не от model abuse,
а от model accident (модель часто пишет `cd /tmp && wget ...` или
`cp out.txt /usr/local/...`).

Урок: sandbox **не привязан к trust level**. Это два независимых
механизма — trust решает «спрашивать ли», sandbox решает «куда
пускать». Связать их вместе — соблазн (мол, скептик == sandbox on,
yolo == sandbox off), но это смешивает «доверие» с «изоляцией».
yolo с sandbox = «я доверяю модели, но не доверяю всему миру»;
скептик без sandbox = «спрошу про каждую команду, но дам ей доступ
ко всему» (имеет смысл когда команды единичные и проверяемые).
Раздельные конфиги дают полную решётку.

## Глава 24 — Аннотация плана: одна доп LLM-ходка вместо угадайки

`load_skill` инструмент работает: модель видит каталог в системном
промпте, может вызвать `load_skill('python')` чтобы подтянуть
инструкции стека. На сильных моделях это норма. qwen-14b — нет:
load_skill пропускается, дальше идёт shell_exec `pytest` без venv,
sandbox отвечает exit 127, цикл retries, fail. Зафиксировано в probe
2026-05-13.

Первая попытка лечения — filesystem detect + keyword detect:
сканировать `pyproject.toml` и т.д., плюс искать в TASKS.md
ключевые слова. Работает в простых случаях, но даёт ложные
срабатывания на спокойных проектах («test» матчится в любой задаче).

Финальный подход — отдельный LLM-проход после плана, перед
выполнением. Аргументы: план (короткий) + каталог скиллов
(короткий) + filesystem hint («обнаружено: python» или «проект
пуст, делаем с нуля»). Запрос: для каждого `## T###:` блока
добавь строку `Skills: ...`. Один turn, температура низкая,
аннотации сохраняются в TASKS.md как часть плана.

При `/go` агент читает аннотацию каждой задачи и грузит указанные
скиллы автоматически перед `code_with_retry`. Модель никогда не
решает про load_skill сама — это делает аннотация-проход с тем
же набором глаз и с явным каталогом. Если пользователь правит
план руками — `/annotate` пере-аннотирует.

Это паттерн «одна LLM-ходка дешевле десяти retry-циклов» в чистом
виде. Доп LLM-вызов после плана стоит секунды; без него цикл
fail/retry на голом pytest съедает минуты и оставляет .venv-мусор.
Тот же приём может пригодиться mid-execution: если модель пошла
делать ту что не было в планах (новый стек), запустить annotation
pass на оставшихся задачах. Пока не реализовано.

## Глава 25 — Plan robustness: верификация в три слоя

Проблема из probe 2026-05-13: задача T001 «Создать структуру»
(Files: setup.py, requirements.txt, weather_cli/) — модель сделала
только папку, отметила задачу done, перешла к T002. План «успешно
выполнен», на диске пусто кроме папки.

Стандартное решение — пайплайн запускает `_run_tests` после
write_file, тесты PASS, делаем вывод что задача готова. Сломанная
логика: тесты успешны потому что **нет тестов** (pytest exit 5
интерпретировали как 0). Модель добавила пустой `assert True` и
пайплайн «удостоверился».

Закрыли тремя слоями верификации в `run_plan`:

1. `_verify_task_files` — каждый путь из `Files:` должен существовать
   на диске. Каталог с `/` проверяется как `is_dir`. Файл без `/` —
   `is_file`. Если хоть один отсутствует → outcome=failed, план
   стопает. Это ловит главный кейс «модель забыла половину файлов».

2. `_verify_task_test_command` — если в TASKS.md `Test command:`
   указана и это не `pytest` (pytest уже погнали через `_run_tests`),
   запускаем команду через `shell_exec yolo` и проверяем exit 0.
   Ловит «файлы есть, но логика битая» когда планировщик заложил
   нетривиальную проверку типа `python setup.py sdist`.

3. Git HEAD validation — модель обязана коммитить в конце каждой
   задачи (правило в mode_code.md). Перед задачей запоминаем `git
   rev-parse HEAD`, после задачи требуем sha != prev. Если HEAD
   не двинулся — задача failed. Это «общая печать» каждого
   завершённого шага, видна в `git log`, перевариваема.

Дополнительно — `auto_git` в `run_plan` создаёт репо и стартовый
`.gitignore` если их нет. Без этого первый коммит модели падал бы
на «no email/name configured». `git config user.email scalpel@local`
ставится локально (не глобально, чтобы не мешать настройкам
пользователя).

Урок: «модель сказала готово» — слабый сигнал. Машинные проверки
дешёвые, дают независимое подтверждение. Где можно проверить
автоматом (файл существует / команда вернула 0 / HEAD сдвинулся) —
надо проверять; где нельзя (текстовые acceptance bullets) —
довольствоваться частичной защитой.

backlog: распространить тот же паттерн на ChoiceCard / future

**10. Где автор не сразу видел архитектурную цену.** Когда
проблема с `queue.py` collision на e2e probe — автор предложил

правки вместо одного abstract patch'а к промпту.

**Meta-урок.** Диалог сильно отличается от code review. На code
review проверяют «работает ли реализация». В диалоге обсуждают
«ту ли реализацию мы вообще задумали». Хорошие архитектурные
решения этой сессии родились из «давай ещё раз посмотрим что мы
хотим» — не из «вот код, найди баг». Когда я предлагал что-то
не лучшее, юзер не правил код — он спрашивал «а зачем». Когда
юзер предлагал что-то не учитывающее side-effect, я не молчал —
показывал side-effect. Получался третий вариант, который часто
оказывался простыр и сильнее обеих исходных идей.

Видимо это и есть «продукт делается в диалоге, не в одиночку».

## Глава 28 — Pending queue вместо синхронного upstream: дёшево и обратимо

Контекст: v0.12 завозит «upstream» — отдельный мощный профиль
(платный API или вторая локальная модель), который мы зовём на
архитектурные развилки. Первая мысль была — синхронный путь:
fork → upstream → answer → builder. Простой, понятный, и
**неправильный для нашего масштаба**.

Почему неправильный. Builder на 14b генерирует 80–120 токенов
в секунду. Upstream — даже на API — это 1.5–3 секунды latency
на запрос плюс время на ответ. На каждый fork в /go это
останавливает агента посреди задачи. На локальном swap'е цена
ещё хуже: LM Studio Auto-Evict перестроит память под другую
модель за 4–7 секунд **в каждую сторону**. Тридцать forks в
одном /go — это пять минут простоя «модель грузится».

Сменили модель так: fork идёт в **pending queue**, builder
получает временный ответ от LocalMetaForker (= 14b в architect
mode) и продолжает работать. Upstream вызывается батчом — либо
в конце /go (`upstream_flush_on=go-end`), либо по явной команде
`/escalate`. Каждый fork upstream подтверждает или возражает; в
случае возражения мы не переписываем код автоматически —
записываем override в FlushSummary с указанием **списка commit
SHA, которые ран при «временной» картине мира**. Юзер потом
сам решает через `/review-overrides` стоит ли откатывать.

Архитектурное обоснование автоматизировать override = нет:
builder мог собрать на временном ответе кучу взаимозависимого
кода. «Переключить решение задним числом» — это рерайт пяти
файлов с потерей рабочего состояния. Лучше показать «вот
commit'ы, где это влияло, человек смотри глазами».

Implementation detail который чуть не похоронил всё: «какие
commit'ы кому атрибутировать». В наивной модели каждый fork
помечает свои commit'ы. Но builder не парсит на каком fork'е
он сейчас работает — TASKS.md не знает про fork-IDs. Мы
выбрали грубый-но-честный подход: **каждый commit во время
жизни очереди атрибутируется ВСЕМ pending forks**. Это
наверняка переоценка scope, но в нашем случае гипероценка
лучше пропуска: юзер при ревью сужает по контексту, агент не
теряет данных. Альтернатива (per-fork scope через
аннотации в TASKS.md) — добавила бы трещину в DSL плана ради
ленивого features.

`/escalate` без аргументов сейчас просто дренит очередь.
Будущее расширение: `--preview` (показать очередь без
flush'а) и `--filter=critical` (только критичные forks).
Делать заранее не стали — пока 30 forks в очереди никто не
видел; первая встреча реального воркфлоу подскажет какие
ручки нужны.

## Глава 29 — Carve-out релиза: когда «доделать всё» хуже, чем «отдать что готово»

Версия v0.12 в плане звучала как «upstream + resume». Обе
фичи про «работа продолжается с накопленного состояния»,
обе крупные. К моменту когда upstream был зелёный (PR-A, B,
C1, C2 — слитые, 1072 теста прошли, три probe'а на реальной
14b показали 12/12, 5/6, e2e-стек работает), resume ещё не
был даже спроектирован толком: STATE.json расширение, entry-
card через ChoiceCard, восстановление _history через compact-
summary — это ещё пара недель работы.

Очевидный соблазн — держать релиз заблокированным, доделать
resume, выкатить «v0.12 — upstream + resume» как и планировали.
Мы так не сделали. Вычленили resume в v0.12.5 — отдельный
релиз, отдельный тег, отдельный § в плане. Аргументы:

1. **Surface fix-isolation.** Если после релиза кто-то
   репортит баг — мы хотим знать в какой фиче. Бандл «upstream
   + resume» делает diff между «у меня сломалась подкачка
   контекста» и «у меня /escalate упал» вычислением вручную.
   Отдельные релизы — отдельные surface'ы. Bisect по тэгам
   v0.12.0 / v0.12.5 / v0.13.0 — это бесплатное наследие.

2. **Завершённость как сигнал команде.** Открытый dev-релиз
   («ещё не доделали resume, но всё остальное работает»)
   психологически даёт право не финализировать. Закрытый
   тег — даёт право двигаться дальше. Это особенно важно при
   автономной работе агента: «доделать всё» расползается в
   рефакторинг, в полировку, в hedge cases которых никто не
   просил.

3. **Тематическая когезия после carve'а.** Когда мы вытащили
   resume в v0.12.5, у v0.12 осталось два трека: upstream
   delegation и debug investigation. Они тоже разные на
   первый взгляд, но оба — про «низкоуровневые слои» (один
   делегирует наверх, другой копает вниз). Прочнее, чем
   «upstream + resume».

Carve-out — это **не отказ от фичи**. Это решение «отдать
готовое сейчас, пока сборка зелёная, а большую вторую фичу
делать с чистой стартовой точкой». В план это записывается
тремя строками: ~~v0.12~~ закрыта, v0.12.5 открыта,
v0.12.5 содержит scope X.

Validation-gate перед carve-решением был **probe-driven**.
Три скрипта в `scripts/`:

- `probe_forks.py` — 4 fork-сценария × 3 формата вывода
  (json/text/structured), 12 проверок. После v0.12 — 12/12
  совпало с ожидаемым.
- `probe_fork_reviewer.py` — 6 ревью-кейсов × 3 температуры,
  главный анти-паттерн: rubber-stamp на override. После
  v0.12 — 5/6 правильно, 0/3 rubber-stamp.
- `probe_e2e_forks.py` — полный /go-стек: detect_forks →
  resolve → annotate TASKS.md → run_plan. После v0.12 —
  decisions block корректно записан, builder-исход (failed
  на write_file пустотой) — pre-existing баг builder'а, не
  v0.12 регрессия.

Это **не unit-тесты**. Это интеграция с живой моделью; мы
платим 2-3 минуты пробега за уверенность что «релиз не
ломает то что в продуктиве». Перед `git tag` — обязательны.
Если probe деградирует — релиз не происходит, проблема
лечится первой.

Метод применим за пределами этого проекта: любой релиз
LLM-системы должен иметь свой эквивалент probe-suite'а на
реальной модели, потому что mocked unit-тесты не ловят
behavioral регрессии (модель раньше всегда отвечала
`structured`, а после промпт-чистки начала ходить через
`text`-fallback — unit-тесты с моками это пропустят).

## Глава 30 — STATE.json для full resume: что писать и что не писать

v0.12.5 — full resume сессии. Контекст: TUI должен подбирать
работу там, где её бросили, после kill / перезагрузки / потери
power. У нас уже был лёгкий resume — `dirty_patch` флаг в
`STATE.json`, при старте всплывает «previous session ended
with an unfinished patch, проверьте `git diff`». Это не resume,
это **напоминалка**. Настоящий resume — продолжить с того
места, в той же фазе, с тем же планом, с открытыми pending
forks.

Дизайн-вопрос: **что сохранять**. Соблазн всегда — «сохрани
всё», и тогда любая ошибка лечится restart'ом. Но «всё» в нашем
случае значит:

- `_history` — ~30k токенов разговора, JSON с tool_calls
  тегами; писать на каждый turn — это I/O-стоимость на горячем
  пути builder'а
- `_loaded_skills` — set имён, копейка по объёму
- `_read_files_history` — список путей, копейка
- pending upstream queue (`UpstreamPendingQueue`) — frozen
  dataclass'ы, дюжину килобайт максимум; ForkOption-кортежи
  не сериализуются в JSON напрямую (нужен mapping)
- состояние Session (токены, cost) — дюжина чисел
- mode / profile / context_limit — пара строк

Сначала пошёл по соблазну — давай всё. Через 20 минут понял:
сохранять `_history` в `STATE.json` на каждом ответе агента —
это amplification I/O в 100×: каждый tool_call = одна-две
строчки логики + полный JSON-dump на 30k токенов. Атомарная
запись через tmp+rename — это ещё syscall на каждое сохранение.

**Принял другое решение: STATE.json хранит только то что**
**нельзя восстановить из других источников.**

- `_history` восстанавливается через compact-summary при
  resume. Это _история разговора_ — для возобновления контекста
  нужна **смысловая выжимка**, а не байт-в-байт реплей. У нас
  уже есть `_compact` который делает summary истории при
  context_limit. Resume этим же механизмом восстанавливает —
  достаточно знать хеш предыдущей истории, чтобы понять «нужен
  ли compact и какой»;
- pending upstream queue — снимок (`open_fork_questions`)
  достаточен: fork_id, question, picker_chosen. ForkOption
  списки опций мы при restore'е не нужны для отображения «N
  forks ждут» — для самого flush'а через UpstreamForker нам
  нужен только question и context, а это всё равно
  идёт через прицельный prompt; ForkOption — клиентский
  кэш, не источник правды;
- `current_task`, `step_phase`, `dirty_patch`, `last_test_status` —
  уже есть, оставляем;
- `mode`, `profile`, `context_limit`, `max_files`, `max_file_lines` —
  конфиг runtime, оставляем (Ctrl+L во время сессии менял
  trust — это надо помнить).

В STATE.json **не пишем**: `_history`, токен-счётчики Session
(восстановим из последнего compact), `_loaded_skills` (восстановим
из task'а), `_read_files_history` (восстановим из _history через
compact). Принцип: STATE.json — мета, не лог.

Дополнительная польза от компактного STATE: **диффы между
сессиями читаются глазами**. Если что-то пошло не так —
`git diff .code-scalpel/STATE.json` после краха показывает что
именно изменилось перед смертью. Полный _history там был бы
шумом.

Эта глава отражает **PR-A** v0.12.5: расширение STATE.json
двумя полями (`history_summary_hash`, `open_fork_questions`)
с pydantic-моделью `PersistedFork` для plain-JSON shape'а.
UI-часть (entry-card «Continue / Restart») и интеграция с
upstream queue — отдельные PR'ы.

## Глава 31 — Save-хуки: где именно писать STATE.json

PR-A v0.12.5 завёз поля в `STATE.json`. PR-B должен ответить
**когда именно** туда писать. Очевидный соблазн — «save после
каждого изменения любого поля». Это амплификация I/O: на
каждый tool call от 14b + один os.replace на tmp-файл. Не
большая стоимость, но и не нулевая, плюс размывает «когда
была crash recovery точка».

Принцип, к которому пришли — **event-driven**, а не
fine-grained:

- save на **task transitions** в run_plan (старт / конец);
- save после **успешного коммита** (поскольку коммит — это
  natural recovery boundary: git уже знает что произошло, и
  STATE может ссылаться на head_after);
- save после **fork enqueue / upstream flush** (изменение
  open_fork_questions);
- save после **compact** (изменение history_summary_hash).

НЕ save'им: на каждом tool result, на каждом stream token, на
каждом read_file. Эти события безопасны для пропуска —
crash в их середине просто откатывает к предыдущей recovery
точке, потери минимальны.

Архитектурный вопрос: **где живёт save**. Два варианта:

1. Централизованный `StateKeeper` класс — owns AgentState +
   методы `mark_task_started`, `mark_task_completed` и т.д.
   StepAgent / Runtime зовут эти методы вместо прямой мутации.
2. Прямая мутация state с явным `_persist_state()` вызовом
   в нужных точках. State — duck-typed обьект, save —
   wrapped в `suppress(Exception)`.

Выбрали (2). StateKeeper класс — это правильное OOP, но
текущий объём — два save'а в run_plan и пара в будущих
PR'ах. Класс на пять методов ради этого — over-engineering.
Прямые мутации более явные («вот тут current_task = task.id,
вот тут save»), и если потом save'ов станет больше — рефактор
в StateKeeper тривиален. Делаем дешёвое решение пока проблема
маленькая.

`_persist_state()` — приватный хелпер в StepAgent. Wrapped в
`suppress(Exception)` потому что STATE — best-effort: crash
в save'е не должен ломать /go. Disk full, permission denied,
sigkill в середине rename — всё это лучше пропустить чем
обвалить плэн.

Эта глава отражает **PR-B** v0.12.5: Runtime → StepAgent
передаёт `state`, run_plan вызывает `_persist_state()` на
старте и конце каждой task. Fork-queue sync и
history_summary_hash — PR-C/D, entry-card на старте TUI —
PR-E.

## Глава 32 — diff для write_file: считаем на стороне tool'а

Параллельно с v0.12.5 наткнулись на старый долг: дифы для
`write_file` мы реконструировали в `agent.py` через synthetic
`Edit(path, "", "")` — пустой search/replace. Это работало
для счёта «модель тронула эти файлы», но не давало
визуального дифа: PatchCard показывал «(no edits)», reviewer
получал текстовое «(no inline diff — write_file based)», а
retry-prompt модели — то же самое плюс полный test output без
указания что именно поменялось.

Картинка из Claude Code (read'er показал) сделала очевидным
что мы недобираем UX: `Update(path) — Added 4 lines, removed
4 lines` + раскраска +/- — это сильно понятнее «вот сюда
смотри». Мы хотим то же самое, и **у нас есть данные** —
мы знаем что было на диске до write'а и что стало после.

Дизайн-вопрос: **где считать diff**. Два места:

1. Reviewer / retry-prompt в `agent.py` — реконструировать
   diff после факта, читая текущее содержимое файла и
   сравнивая с... чем? У нас нет «снимка до». Это уже
   потеряно к моменту когда reviewer хочет смотреть.
2. Сам tool в `tools/agent_tools.py` — там есть **обе
   версии** в одном функциональном кадре: `old_content =
   target.read_text()` уже читается для replace/insert
   режимов, надо только дотянуть это до overwrite-режима и
   распаковать в `difflib.unified_diff`.

Выбор очевиден — (2). Принцип «считать данные ближе к
источнику» работает не только для производительности, но и
для **семантической точности**. Reviewer не должен играть в
угадай-кто-был-до — это уровень абстракции выше, чем у него
есть.

Имплементация — `ToolResult` получает опциональное поле
`diff: str | None`. `_tool_write_file` после успешной записи
вызывает `_unified_diff_for_write(path, old_content,
final_content)`, кладёт в результат. Новые файлы получают
`/dev/null` как `from`-label — стандартный unix-маркер
«брэнд новый». 5 новых юнит-тестов: new file / overwrite /
replace range / insert / failed (diff = None).

`agent.py` получает helper `_collect_write_file_diffs(tool_results)`,
который склеивает все диффы из tool_results текущего turn'а.
В reviewer'е и retry-prompt он подменяет старый fallback
«(no inline diff — write_file based)» — теперь там реальный
unified-diff.

Что **не** сделали в этом PR (отложено в бэклог): TUI-карта
`UpdateCard` с цветной раскраской и заголовком «+N −M». Это
требует прокидывать `tool_results` в `PatchAttempt` или
передавать `step_result` в `_render_attempt` — рефактор
средний. Данные на руках, проблема только в рендере; когда
дойдём до UX-полировки серии — поднимем.

Урок для следующих tool'ов: если tool изменяет файл,
**возвращать diff из tool'а же**. Не отдавать reconstruction
работу в downstream'е — там данные хуже.

## Глава 33 — Probe-suite v2: что и почему именно сейчас

К концу v0.12 мы догнали: старые probe'ы (`probe_forks`,
`probe_fork_reviewer`, `probe_e2e_forks`, `probe_code`,
`probe_recipes`) дают точечные ответы — fork API даёт 12/12,
reviewer не штампует, structured output вылавливает кейсы
которых не вытягивает text. Это знание **полезно**, но
узкое. Оно говорит: «отдельные tool-каналы рабочие».
Не говорит: **сколько живых трудозатрат стоит решить
типовую задачу через scalpel**.

### Зачем именно сейчас

Три причины собрались в одну точку:

1. **Проект близок к финалу.** v0.12 закрыт, v0.12.5
   доделывается, после v0.13 (knowledge layer) серия статей
   садится за дело. Это **последняя возможность собрать живые
   данные пока решения свежи в голове** — кто что менял, зачем,
   какие были альтернативы. Через полгода эта информация
   разбежится.

2. **Свежие PR'ы не валидированы behaviorally.** v0.12 PR-C2
   (upstream queue), v0.12.5 PR-A/B (STATE.json + save-hooks),
   side-PR (ToolResult.diff) проверены только unit-ами. На
   14b они могут регрессировать: длинный diff в retry-prompt
   раздувает context, save-хуки могут задеть горячий путь /go.
   Юнит-тесты этого не ловят — нужен живой прогон.

3. **Серия статей требует материала.** Каждая из 6 запланированных
   статей упирается в конкретные иллюстрации:
   - Статья 1 (вирусная): «вот я за 5 turns починил баг» — с реальным `chat.jsonl`.
   - Статья 4 (hallucinations): «вот scalpel уверенно соврал про
     метод, потому что в MAP'е не было docstring'а» — конкретный
     payload с тэга где это происходило.
   - Статья 5 (benchmark): timeline pass-rate × версия.
   12/12 на forks этих иллюстраций не даст.

### Чем это НЕ является

Это **не QA-проход** в моменте. Если scalpel на 14b споткнётся
посреди прогона — я **не лезу чинить код**. Записываю, веду
диалог дальше как живой юзер, после прогона разбираю в
`evaluation.md`: где было плохо, **почему оно было плохо**
(гипотеза о причинах), **как можно было бы попытаться лечить**
(теоретически — без обязательства реализации), и **что это
говорит о принятых архитектурных решениях** (верны ли были).

Фиксы — отдельный пайплайн после серии прогонов. Мы собираем
**диагностику**, не правки.

### Что ищем

Иерархия метрик:

**Первичная**: **решена ли задача в принципе**. Binary.
Pytest exit 0, или TASKS.md существует и валиден, или новый
тест проходит и старые не сломались — в зависимости от
сценария. Если задача не решена — это **уже сильный сигнал**
сам по себе, дальнейшие числа второстепенны.

**Вторичная** (если задача решена): **трудозатраты**:

- сколько turn'ов мне как юзеру понадобилось,
- сколько LLM round-trips потратил scalpel,
- сколько токенов суммарно (prompt + completion),
- сколько retry в `code_with_retry`,
- сколько tool calls (по типам),
- wall time.

Сравнение между версиями: задача решилась везде / только с v0.X /
не решилась нигде. Если решилась везде — кривая трудозатрат
показывает где мы реально стали эффективнее.

Сравнивается **между версиями** (тэги от MVP до main) и
**между типами задач** (планирование диалогом / по ТЗ / фикс
бага / новая фича) **на разных размерах проекта** (mini-cli /
mini-fullstack).

Сигналы которые могут вылезти:
- регрессия pass-rate на свежем PR'е → ещё горячее можно
  откатить;
- линейный рост tokens на типовой задаче от v0.5 к v0.10 →
  понимаем сколько средняя задача «стоит» сейчас vs тогда;
- падение retries после v0.9 (machine guards) — подтверждаем
  нарратив статьи 6 (архитектура);
- появление новой возможности в v0.X (например, write_file
  с v0.6) видно как новый столбец в capabilities-матрице.

### Почему живой я-юзер, не auto

LLM в роли юзера шумит: при temperature 0 она даёт одни
ответы, при 0.5 — другие, между прогонами variance
непредсказуем. На метрике «трудозатрат» это смертельно — мы
не сможем отличить «scalpel улучшился» от «юзер-LLM
случайно лучше формулировал».

Скриптованный юзер с зашитыми репликами решает variance, но
рушит главное — реалистичность. Если scalpel задал
неожиданный вопрос, скрипт ответит «не в тему», диалог
ломается, метрика теряет смысл.

Living user — я (Claude) — даёт оба свойства: адаптируюсь к
тому что scalpel выдал, остаюсь в характерной речи юзера
(`user_tone_of_voice.md` собран из истории живых сессий),
параллельно отслеживаю качество и пишу `evaluation.md` после
прогона. Один источник истины для двух функций — управление
диалогом и оценка.

### Артефакты как первый класс

Каждый прогон — папка `docs/article/probe-runs/<run-id>/` в
репо с фиксированной структурой: `meta.json` (всё для
reproducibility), `chat.jsonl`, `tools.jsonl`,
`fixture_initial.tar.gz`, `final_tree/`, `metrics.json`,
`verdict.json`, `evaluation.md`, `agent_plan.md`,
`user_plan.md`, `timing.json`, `notes.md`. Если для прогона
артефакт неприменим — лежит заглушка с пометкой почему.

Это даёт **аудит-trail на полгода вперёд**. Когда сядем за
статью 4 и захотим показать конкретный hallucination — не
будем выдумывать пример, откроем `probe-runs/v0.6-some-id/`
и процитируем строки `tools.jsonl`.

Протокол прогона зафиксирован в
[`scripts/probes_v2/PROTOCOL.md`](../../scripts/probes_v2/PROTOCOL.md) —
от старта (`probe start`) через цикл реплик (`probe step`)
до финализации (`probe finalize`). Реализация runner'а —
следующим PR'ом.

## Глава 34 — Methodological lapse: probe мерил половину workflow

Probe-suite v2 заработал, и я кинулся гонять — pilot на main,
historical-серия по тэгам. Через 8 прогонов остановили: я везде
шёл `--mode ask` → `--mode code` и финализировал. Команду
`probe go` (которая зовёт `agent.run_plan` — то самое **что и
делает реальную работу** в scalpel'е) **ни разу не вызвал**.

Получалось: ask-мode отвечает «расскажу как сделать», code-mode
крутит `code_with_retry` на один shot (потому что
`iterative_patch_loop=False` по умолчанию + я не передавал
`force_loop`), модель пишет fenced ```python``` блоки для
человека, applier ничего не применяет, файлы не появляются.
**Это не результат «инструмент не работает».** Это «я мерил
половину пайплайна».

Канонический workflow в scalpel'е (повторяет TUI):

1. `/mode plan` — переключение в plan mode, который через
   `_PLAN_MODE_ADDENDUM` учит модель писать **TASKS.md в нашем
   DSL** (`## T001:` заголовки + Files/Acceptance/Test command)
2. Юзер пишет задачу — модель отвечает планом
3. `agent._maybe_save_plan(reply)` сканирует regex'ом `## T\d{3}:`
   и **автоматически** сохраняет `.code-scalpel/TASKS.md`
4. `/loop` (toggle iterative_patch_loop) — опционально, default off
5. `/go` (на v0.6+; на v0.3-v0.5 называется `/run`) — запуск
   `agent.run_plan()` который ходит по TASKS.md через
   `code_with_retry`

Я этот workflow знал по коду, но в probe-runner'е **прыгнул**
этап plan'а — сразу просил «спроектируй и реализуй» в ask
mode, потом «сохрани файлы» в code mode. Модель в этих
mode'ах не имеет инструкции «писать TASKS.md в DSL» — она
честно отвечает текстом юзеру.

Урок не про методологию probe. Урок **общий**:

> Прежде чем сравнивать «что инструмент умеет на разных версиях»,
> убедись что **сам инструмент дёргается правильно**. Иначе ты
> сравниваешь свою методологическую ошибку с самой собой.

Это **первый класс ошибки** в эмпирическом исследовании
LLM-инструментов: путать «модель не справилась» с «я не дал ей
шанс». Чтобы такого не повторять, в `PROTOCOL.md` появилась
секция «Семантика mode'ов — реплики юзера»: ask просит «расскажи»,
plan просит «составь TASKS.md», code просит «реализуй»,

## Глава 35 — Historical series: что меняется от тэга к тэгу

probe-suite v2 — **не для показа LLM-ошибок**, а для **видения

- v0.8 → v0.9: «то же качество дешевле» — это видно только

- v0.10 → main: «слепой регресс 4 версии подряд» — без widerscope

- Fork-эпоха: «новые узкие capabilities + регресс broader осей»

Если бы probe-suite v2 был в CI с самого начала, **регресс
v0.10 был бы пойман до merge'a** PR `feat/fork-api`. И это —
не теоретическая польза, а **конкретный пропущенный момент**
