# Критика черновика статьи и план переработки

## Общее впечатление

Черновик сильный по материалу, но сейчас он читается как длинный devlog: последовательность инженерных заметок по разработке `code-scalpel`.

Это не плохо, но для Хабра такой формат рискованный: читатель быстро теряет главную линию. В тексте много хороших открытий, но они конкурируют друг с другом.

Главная задача переработки — превратить лог разработки в статью с одной сильной идеей.

## Центральная мысль статьи

Нужно держать фокус вокруг тезиса:

> Для weak local LLM главная проблема не “интеллект модели”, а интерфейс между моделью, кодом, контекстом и инструментами.

Или короче:

> Слабую LLM можно усилить не размером модели, а правильно спроектированной средой вокруг неё.

Вся статья должна работать на этот тезис.

## Что в черновике уже хорошо

### 1. Есть настоящая инженерная история

В тексте много цепочек вида:

- думали X;
- проверили;
- получили неожиданный результат;
- поменяли архитектуру;
- сделали вывод.

Это сильнее обычного “гайда по LLM”.

Особенно хороши главы про:

- unified diff vs SEARCH/REPLACE;
- context budget;
- tool calling;
- hallucinations;
- model benchmark;
- mem0 spike;
- duplicated execution path;
- read_file как window/find;
- post-hoc guard.

### 2. Есть цифры

Нужно сохранить:

- 12/15 на unified diff;
- 13/15 после fuzzy fallback;
- 23/24 после native function calling;
- 24/24 у Gemma;
- 23/24 у qwen2.5-coder-14b;
- 5–8k токенов до оптимизации;
- ~200 токенов после перехода к lazy context;
- падение benchmark после перестройки context structure.

Цифры делают текст убедительным.

### 3. Есть фейлы

Но сейчас они описаны слишком спокойно. Нужно усилить ощущение реальной разработки: “мы ошиблись”, “мы пошли не туда”, “мы потеряли время”, “починили не там”.

## Главная проблема черновика

Сейчас это “всё, что произошло”.

А нужно сделать “самое важное, что мы поняли”.

Не надо пытаться вставить все главы в одну статью. Лучше сделать первую сильную статью, а остальное оставить для серии.

## Рекомендуемый формат первой статьи

Рабочий заголовок:

> Мы делали coding-агента для слабой локальной LLM — и оказалось, что проблема не в модели

Альтернативы:

> Почему слабым LLM нужен не лучший промпт, а нормальная инженерная среда

> Как мы строили coding-агента для qwen2.5-coder-14b и нашли проблемы не там, где ожидали

## Предлагаемая структура

### 1. Hook

Начать не с обзора фреймворков, а с сильного факта.

Например:

> Мы хотели сделать простого coding-агента для qwen2.5-coder-14b.
> Думали, что главная проблема будет в качестве модели.
> Оказалось — в формате diff'а, структуре контекста и том, какие инструменты модель получает.

Или:

> Первый benchmark показал 12/15. Модель понимала задачу, но ломала patch.
> Проблема была не в коде, а в формате общения между LLM и инструментом.

### 2. Зачем вообще писать своего агента

Коротко, без длинного списка всех фреймворков.

Смысл:

- готовые инструменты часто рассчитаны на сильные frontier-модели;
- локальные 7B–14B модели требуют другого UX;
- нельзя просто дать модели весь проект и попросить “сделай хорошо”;
- нужен маленький, supervised, patch-oriented агент.

Список Letta/OpenHands/LangGraph можно сильно сократить или вынести в отдельную вставку.

### 3. Первый большой фейл: unified diff

Это один из лучших фрагментов.

Нужно сделать его центральной сценой:

- сделали benchmark;
- получили 12/15;
- все ошибки были не в понимании задачи, а в формате patch;
- unified diff требует слишком много инвариантов;
- SEARCH/REPLACE оказался лучше, потому что ближе к тому, как LLM копирует текст.

Нужно показать пример:

```diff
@@ -0,0 +N @@
````

и рядом SEARCH/REPLACE.

Вывод:

> Формат взаимодействия с моделью иногда важнее самой модели.

### 4. Второй фейл: контекст раздувался до абсурда

Показать “до/после”:

Было:

```text
system prompt + список файлов + первые файлы целиком = 5–8k токенов даже на “привет”
```

Стало:

```text
overview проекта + tool calls по требованию = ~200 токенов на простой запрос
```

Главная мысль:

> Контекст — это не мешок, куда надо напихать побольше. Это бюджет, которым надо управлять.

Нужно добавить схему:

```text
Before:
User request
  ↓
Full project map + files
  ↓
LLM

After:
User request
  ↓
Small overview
  ↓
LLM calls tools:
  - map_file
  - read_file
  - grep
  - goto_definition
```

### 5. Третий фейл: hallucinations лечатся не промптом

Очень сильный блок.

Нужно показать конкретный пример:

* модель уверенно сказала, что `mark_compacted` делает compact;
* метод реальный;
* класс реальный;
* но смысл выдуман.

Это важно назвать:

> Это не fabrication, а misattribution.

Вывод:

> Если модели не хватает информации в project map, промпт-правила не спасут. Надо обогащать данные: docstrings, imports, связи между файлами.

Отдельно показать post-hoc guard:

* prompt rule не заставил модель читать файл;
* добавили hard constraint в коде агента;
* если модель показывает код без `read_file`, ответ отклоняется и запускается retry.

Вывод:

> Prompt is soft constraint. Guard in code is hard constraint.

### 6. Benchmark моделей

Оставить таблицу, но сократить объяснение.

Главные акценты:

* Gemma general model неожиданно взяла 24/24;
* qwen2.5-coder-14b остался лучшим Pareto-вариантом из-за скорости;
* большие MoE Qwen оказались хуже;
* размер модели не гарантирует качество.

Вывод:

> Модель надо выбирать не по размеру и не по marketing benchmark, а по своей задаче и своему harness.

### 7. Параллельные агенты

Это можно оставить как короткий раздел ближе к концу.

Не надо подробно перечислять все 7 агентов. Лучше показать вывод:

* research отдельно;
* implementation отдельно;
* audit отдельно;
* файловые границы обязательны;
* без границ агенты начинают конфликтовать.

Хорошая формулировка:

> Параллельные агенты — это не “пусть ИИ пишет за меня”, а разделение труда: один пишет, второй проверяет, третий исследует, четвёртый ищет coverage gaps.

### 8. Финальный вывод

Финал должен быть сильным.

Например:

> Мы начинали с идеи “сделать агента для слабой модели”.
> В итоге оказалось, что агент — это не модель.
> Агент — это контекстный бюджет, форматы обмена, tool loop, guards, benchmark и UX, который не даёт модели разрушить проект.

Или:

> Weak LLM не надо просить быть умнее. Ей надо дать среду, в которой правильное действие дешевле неправильного.

## Тон статьи

Нужен тон:

* инженерный;
* честный;
* без AI-hype;
* с самоиронией;
* без корпоративного PR.

Не надо писать:

> Мы разработали инновационный инструмент...

Лучше:

> Мы хотели маленький локальный coding-agent. По дороге выяснилось, что почти всё ломается не там, где ожидали.

## Главный риск

Не превратить статью в changelog.

Каждая глава должна отвечать на вопрос:

> Какой общий инженерный урок из этого следует?

Если урока нет — фрагмент убрать или вынести в отдельную статью.

## Итоговая цель переработки

Сделать из черновика не “историю всех коммитов”, а статью о том, как проектирование среды вокруг LLM оказалось важнее самой LLM.

Главная формула:

> Weak LLM + хороший tool loop + правильный формат patch'ей + управляемый контекст + hard guards + benchmark
> лучше, чем
> большая модель + длинный промпт + надежда.
