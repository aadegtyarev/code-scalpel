## Главное наблюдение

**Разрыв между «узкий fix» и «широкая создание»**:
- probe_code (calc.add fix через SR) — работает: ✓ 1 attempt
- notes_cli «создай 3 файла с нуля» — НЕ работает: 0 SR, 0 tools

Хотя на v0.7 уже есть **и** SR-pipeline (probe_code это
доказывает), **и** `write_file` tool, и `project_map`, и
`on_tool_executed` hook — **системник не пушит** к этим
инструментам на задачу «с нуля». Модель честно пишет fenced
```python``` для человека, как в v0.3-v0.4.

## Гипотезы о причинах

- **Системник v0.7 учит «исправь точечно через SR»**, но **не учит
  «создавай файлы через write_file»**. Это **два разных интента**,
  и системник освоил только первый.
- Hard-rule «enforce read before show» / project_map-first
  ещё не landed (это v0.9-v0.10 по моим оценкам).
- 14b без явного «при создании нового — write_file» предпочитает
  объяснять человеку.

## Архитектурный smell-check

**Это переломный момент в анализе.** До v0.7 я предполагал
«scalpel станет L3 как только write_file появится». **Не стал.**
Появление tool в API ≠ модель пользуется. Системник должен
явно научить **для каждого жанра задачи**:
- «фикс багa? → SR через ask/code» (work)
- «новый проект? → write_file через code» (not work на v0.7)

Это **прямое доказательство** урока главы 6 девлога: «обогащай
данные, не промпт». Здесь данные (=tool exists) есть, но
**промпт-обвязка** под create-flow не подтянута.
