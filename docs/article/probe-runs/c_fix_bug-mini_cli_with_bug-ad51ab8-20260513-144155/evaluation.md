# Evaluation: c_fix_bug-mini_cli_with_bug-ad51ab8-20260513-144155

## One-liner

14b нашёл баг с 1-го turn'а, но при патче через write_file сломал
core.py (def методы вне класса), 3 retry усугубили; сдался на
3-м user-turn'е, 140k prompt tokens, 0 commits.

## Trajectory

- **Turn 1** «pytest падает, разберись пожалуйста. недавно правил
  core.py, может там» — scalpel за один turn сделал project_map →
  read_file core.py → read_file test_core.py → run_tests → ещё 2
  read_file. Точно увидел упавший тест `test_mark_done_flips_flag`,
  предположил «давай посмотрим на метод mark_done». **Диагностика
  правильная и быстрая**.
- **Turn 2** «ну ок, нашёл. почини и закоммить» — `write_file
  mini_cli/core.py` с полным новым содержимым. Diff показал
  катастрофу: пропали module docstring и docstring TodoStore,
  пропал `@dataclass` декоратор у Todo, **все методы def оказались
  без отступа** (вне класса). Запустил pytest — упало (`TypeError:
  Todo() takes no arguments` и куча других). Прочитал свой
  собственный битый файл, увидел отсутствие `add` (хотя на самом
  деле просто не на своём месте), занял позицию «давай чинить».
- **Turn 3** «ты файл поломал, почини полностью» — два
  дополнительных `write_file` + 1 run_tests. Каждый retry
  усугублял: появился двойник `mark_done` (один внутри TodoStore,
  один снаружи), `def __init__` без отступа, дубли. На третьем
  write_file модель «зациклилась» — повторяла «Извините за
  путаницу, давай починим mark_done и add» дважды подряд без
  новых tool-calls. Я закрыл прогон.

## Хорошо (наблюдения, не правки)

- **Диагностика за 6 tool-calls** — project_map, read_file, run_tests,
  снова read_file. Минимально-инвазивно, точно. Не нужно было
  никаких подсказок от меня.
- **Понимание pytest-трассы** — увидел assertion error, дошёл до
  правильной строки `fresh[0].done is True`, связал с реализацией
  `mark_done`. Логика модели работает.

## Плохо (наблюдения, не правки)

- **write_file целиком ломает большие файлы**. core.py = 60
  строк, не гигантский, но 14b теряет:
  - вертикальную структуру (отступы методов класса);
  - декораторы (`@dataclass` пропал);
  - docstring'и (модуля + класса);
  - даже структуру «методов нет вне класса».
- **Diff'ы видны (наш свежий `ToolResult.diff` работает)**, но
  scalpel их не использует для self-критики. Он пишет файл,
  получает diff, но не сверяет «получилось ли то что я хотел».
  Pytest даёт ему обратную связь, но только после.
- **3-кратный retry без улучшения**: каждое следующее
  `write_file` усугубляло. Модель не имеет механизма откатить
  плохой patch — продолжает писать на сломанный код.
- **Loop-detection не сработал**: к третьему turn'у scalpel
  буквально повторил себя слово в слово («Прошу прощения за
  повторные вызовы. Давай починим...» — два раза подряд). Семантик
  anti-loop из v0.9 это не поймал.
- **140k prompt tokens на 3 user-turn'а**. По 47k на turn в
  среднем. Это огромно для задачи в 60-строчном файле. На
  длинных задачах быстро упрётся в context_limit.

## Гипотезы о причинах

- **Главная**: `write_file content=<полный файл>` — слишком
  большой ход для 14b на сложных файлах. Модель копирует не
  построчно, а «перепридумывает» структуру по памяти, теряя
  детали.
- **Дополнительная**: после первого плохого `write_file` scalpel
  читает свой собственный битый файл, и **видит его как
  «правильный»** (тренд авто-confirm bias). Доверяет своему
  предыдущему ходу больше чем здравому смыслу.
- **Усиливающее**: нет автоматической post-write валидации
  (`py_compile` / smoke-test) — pytest ловит только runtime
  ошибки, не структурные.

## Как теоретически можно было бы лечить

1. **Подталкивать к узким write_file**. Для одной функции —
   range-based (`start_line/end_line`), не целиком. Это в
   prompt'е system.md: «если меняешь одну функцию, используй
   start_line/end_line, не overwrite». Будет ли 14b слушать —
   надо probe'ом проверять.
2. **Post-write валидация на стороне tool'а**. Если
   `write_file` записал python-файл, запустить `python -m
   py_compile` — если синтаксис битый, откатить и вернуть ошибку.
   Это **гарантия что write_file не оставит проект в неработающем
   виде**. Аналог git pre-commit hook, но на уровне tool.
3. **Diff-aware self-check**. У нас уже есть `ToolResult.diff` —
   можно научить scalpel смотреть на размер diff после
   write_file: если ожидал «изменить mark_done» а diff показал
   `-44 +33`, это сигнал «удалил больше чем добавил, проверь».
4. **Жёсткий лимит retry на write_file без прогресса**. После 2
   write_file с pytest fail — стоп, прокинуть upstream или
   позвать юзера. У нас есть `debug_pass_max_attempts=2` для
   тестов, аналогичное должно быть для патчей.

## Архитектурный smell-check

- **`write_file` как «универсальный» tool с тремя режимами** —
  идея хорошая (минимум схемы для модели), но на практике 14b
  всегда выбирает overwrite-режим даже когда надо range. Возможно
  стоит **намеренно разделить на 2 tool'а**: `write_file_new`
  (только создание) и `update_file` (range/insert). Это
  принуждает модель явно выбирать интент. Backlog-пункт у нас
  уже есть в plan.md (UpdateCard, но идея в той же плоскости).
- **Channel-unification работает** — `Session.prepare_turn`
  пропустил мою реплику в нужном формате, scalpel не сорвался в
  «explain in abstract» режим. Глава 17 девлога подтверждается.
- **ToolResult.diff даёт нужные данные** — мы видим что именно
  scalpel испортил в первом write_file. Без этого поля
  диагностика была бы намного сложнее. v0.12.5 side-PR
  оправдывает себя.
- **Иерархия метрик** работает: задача **не решена** (binary 0).
  Вторичные числа — трудозатраты 140k tokens / 12 tool calls /
  3 user-turns — показывают **сколько ушло на тщетные попытки**.

## Трудозатраты (субъективно)

3 user-turns. По плану до 20. Я мог бы продолжать дальше —
наверняка дотолкал бы scalpel до починки за ещё 5-7 turn'ов
точечными подсказками («скопируй файл целиком из памяти», «не
делай overwrite, делай range»). Но **это был бы не probe**, а
«я починил руками через посредника». Финализировал на
`user_gave_up` потому что **14b без подсказок не справился**, а
это и есть результат который мы хотели измерить.

Wall time 6 минут, я хотел бы видеть это под 2 минуты на простой
задаче. 47k prompt tokens / turn — это очень дорого.

## Verdict

**2/5**. Диагностика — 5/5, исполнение — 1/5. Если бы scalpel
просто **сказал** «нужно добавить `self._write(items)` после
строки `item.done = True`», задача была бы решена. Но он
обязательно пытается **сам сделать write_file** и убивает на
этом проект.

- **plan.md backlog**: рассмотреть разделение `write_file` на
  два tool'а (`write_file_new` / `update_file`) или хотя бы
