## Гипотезы о причинах

- **code_with_retry на v0.5 — каркас без системника**: метод
  существует, но pre-prompt instructions «эмитить SR» в `system.md`
  ещё не добавлены. Это **разрыв** между «архитектурно есть
  возможность» и «модель пользуется».
- Системник продолжает быть generic «расскажи как сделать»,
  без edit-формата. Модель честно отвечает в режиме «расскажи
  человеку», без попытки эмитить applier-friendly формат.

## Архитектурный smell-check

**Третья точка baseline-кривой** (v0.3 / v0.4 / v0.5) — все L1,
все 0 tools, все 0 SR. Это **сильный signal**: появление
архитектурного кода (`code_with_retry`) **недостаточно** — нужен
системник который учит модель формату. Без этого 14b пишет
fenced для человека.

Это прямое подтверждение тезиса главы 6 девлога: «промпт-правило
не лечит — обогащай данные». В случае v0.5 правило (mode=code +
вызов code_with_retry) есть, но **системник не учит формату** —
поэтому модель не делает SR.

Ожидаем что L2 появится на тэге где landed «SR-формат описан в
prompts/system.md». Это commit 3020bfb (Swap unified diff for
SEARCH/REPLACE blocks) — попадает в v0.5.0 (мы тестируем после
него), значит **частично уже там**. Но видимо системник для
этой версии scalpel ещё не обновлён под SR.

**Это первая регрессия в серии.** Live прогон не показал
изменений (L1 как был), но детерминированные probe'ы видят
просадку. Что могло измениться v0.4 → v0.5:
- появился `code_with_retry` (commit a06e0c9, попал в v0.5)
- видимо изменился системник под новый flow
- видимо отключился старый «прямой» путь который проходил тесты

`probe_code` на v0.5 не справился с фиксом `calc.add` — модель
сделала patch, applier откатил («rollback ran»). Значит SR-блоки
**модель уже эмитит** на v0.5, но они **некорректные**
(applier не применяет). Это и есть «начало» SR-pipeline'а — есть
попытка, но не работает.

Это сильный сигнал: между v0.4 и v0.5 произошёл **архитектурный
сдвиг с регрессией по числам**. Не «улучшение по нарастающей»,
а **переходный кризис**, типичный для крупного рефакторинга.
