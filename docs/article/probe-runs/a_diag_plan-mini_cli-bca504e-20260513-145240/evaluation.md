# Evaluation: a_diag_plan-mini_cli-bca504e-20260513-145240

## One-liner

TASKS.md создан только после явной подсказки про write_file (turn 3), проект не читал, формат — свободный markdown не наш DSL; 3 turn'а, 23k токенов, 1 tool call.

## Trajectory

- **Turn 1** «хочу ярлыки на задачах, как сделать?» — scalpel ответил textbook-стилем, расписал подход с `add_task("...", ["важное"])`, `list_tasks_with_filter` etc. **Полностью игнорировал реальный проект** — ни project_map, ни read_file. Придумал имена функций из головы (`add_task`, `list_tasks_with_filter` — у нас этих имён нет). 0 tool calls.
- **Turn 2** «ну ты собственно сделай, разложи в .code-scalpel/TASKS.md» — снова **0 tool calls**. Только напечатал markdown-список задач #1..#7 в text reply. Файл на диске не появился.
- **Turn 3** (contamination — я прямо подсказал «через write_file») «ты что не можешь сам файл сделать?» — наконец сделал `write_file .code-scalpel/TASKS.md`. Контент скопировал из turn 2 как есть.

## Хорошо (наблюдения, не правки)

- **Дёшево по токенам**: 23k за весь прогон vs 140k в probe #1. Когда не лезет в tool-loop, расход умеренный.
- **Список задач не глупый по содержанию** — 7 шагов: добавить поле labels, парсер ярлыков, фильтр, обновить list, обновить cli. Если переписать в наш DSL — это валидный план.
- **Быстрый text-reply** (под минуту wall-time на turn) — для чистого explain-режима ОК.

## Плохо (наблюдения, не правки)

- **Не прочитал проект ни разу до создания TASKS.md**. На turn 2 уже надо было хотя бы project_map дёрнуть — у нас `mini_cli/core.py` с классом `TodoStore`, а scalpel пишет «найти определение функции `add_task`» (которой нет). План оторван от реальности.
- **TASKS.md в свободной форме, не в нашем DSL**. Ожидаем `## [_] T001: <title>\nFiles: ...\nTest command: ...\nAcceptance: ...`, получили `## Задача 1: ...\n- Описание: ...\n- Этапы: ...`. Наши мехчекеры (`tasks_have_required_fields`) это не примут, /go не запустит план.
- **Не использует write_file пока я прямо не попросил**. В ask mode дефолт — text-explanation, инструменты не дёргает. Для сценариев планирования это значит надо либо менять mode, либо учить ask-системник «если просят сделать — делай».
- **Contamination в turn 3**: я подсказал «через write_file». Без меня scalpel не догадался — даже на «сделай файл» отвечал markdown'ом в reply, не tool-call'ом. Это сильный сигнал.

## Гипотезы о причинах

- **Системник для ask mode не пушит в tool-use**. Сравни probe #1 — там тот же 14b с тем же системником **сразу шёл в project_map + run_tests**. Разница в формулировке user-запроса: «pytest падает» — явный triggering action, «как сделать ярлыки» — приглашение объяснить. 14b читает intent и выбирает поведение, не имея чёткого правила «при /plan делай TASKS.md в нашем DSL».
- **Plan-mode у нас отдельный**. `/plan` slash в TUI наверняка дёргает другой system addendum («ты составляешь план в нашем DSL»). Probe идёт через `ask` — нет этого addendum'а.
- **DSL TASKS.md модель не знает**. Никаких few-shot примеров нашего формата в `system.md` нет. Модель «изобретает» формат каждый раз.

## Как теоретически можно было бы лечить

1. **Дать `a_diag_plan` сценарию `mode=plan`** — в runner добавить параметр выбора mode. Тогда `ask` для probe #1 (debug сценарий), `plan` для probe #2 (planning сценарий). Это **правильный путь** — мы не хотим переучивать ask делать всё.
2. **Few-shot примеры TASKS.md в plan-системнике**. Сейчас нет. Добавить 1-2 коротких блока «вот как выглядит наш план» прямо в `prompts/mode_plan.md` (если такого нет — завести).
3. **Hard-rule на project_map**. По аналогии с post-hoc guard'ом для read_file (глава 14 девлога): «если генерируешь TASKS.md и не вызвал project_map в этой сессии — reject reply». Жёстко, но 14b не угадывает имена файлов.
4. **Mode-validator на TASKS.md**. Не запускать /go без проверки DSL формата. Это и сейчас есть частично через `parse_tasks_md` — но он молча возвращает []. Должен ругаться явно.

## Архитектурный smell-check

- **`mode=ask` для планирования — неподходящий выбор**. Это не баг архитектуры — это **отсутствие отдельного plan-mode в probe-сценариях**. Сценарий `a_diag_plan` должен запускаться в `/plan` режиме TUI, не в `/ask`. Probe-runner это не моделирует.
- **DSL TASKS.md без few-shot** — известный риск. У нас в `system.md` отлично объяснены edit-формат (SEARCH/REPLACE) и tool-use, но **не объяснён формат плана**. Модель его «знает» только из контекста, когда видит существующий TASKS.md.
- **Channel-unification** работает — реплики уходят через `Session.prepare_turn`. Стиль ответа модель меняет под mode, не под канал. Глава 17 девлога подтверждается ещё раз.
- **0 tool calls в turn 1+2** — это **архитектурно**: ask-системник не имеет hard-rule «при просьбе сделать → tool». Он мягко подталкивает («Вот мои инструменты, используй когда надо»). Для 14b это слишком soft — модель в режиме «объясняю» не помнит про инструменты.

## Трудозатраты (субъективно)

3 user-turns, 234 сек wall time, 23k токенов. По сравнению с probe #1 — в **6 раз дешевле**. Но и без contamination probe скорее всего не дошёл бы до TASKS.md вообще. На третьем turn я уже подсказал — это не «решение задачи», а «решение задачи с моей помощью».

Если бы я был обычным юзером и не знал про write_file — после turn 2 я бы решил что «он только объясняет, файлы не создаёт» и пошёл бы делать руками. То есть **default-experience плохой**.

## Verdict

**2/5 по живому ощущению**. Текстовый план разумный (если переписать в наш DSL), но scalpel:
- не читал проект,
- использовал инструменты только после явного запроса,
- сгенерировал TASKS.md в чужом формате.

По мехчекерам `tasks_md_present` = True, `tasks_count_ge_3` = True
(7 задач), но `tasks_have_required_fields` и `paths_valid` **бы не
прошли** при строгой проверке. Pass-критерии мехчекеров в этом
прогоне не реализованы как код — только декларированы в
scenario.md. Это TODO для следующего PR.
