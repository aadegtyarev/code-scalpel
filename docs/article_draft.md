# Проектируем AI-агента вместе с Claude: от идеи до архитектуры

*Не «как использовать LLM», а как выглядит настоящий co-design на практике.*

---

## Откуда идея

Хотел инструмент для работы с кодом через слабые локальные модели — qwen2.5-coder-14b в LM Studio. Не замену Claude Code, а что-то скромное: маленькая задача, минимальный патч, тест, дальше. Назвал `code-scalpel`.

Вместо того чтобы сразу писать код — решил сначала спроектировать в диалоге с Claude. Получилось интереснее чем ожидал.

---

## Как выглядел процесс

Не «скажи что делать» и не «сделай за меня». Скорее: я приношу идею, Claude уточняет, оба критикуем, решение кристаллизуется.

Первый вопрос от Claude был не про стек, а про ограничения: какая целевая модель, что точно не нужно в MVP. Это определяет дизайн сильнее чем список фич.

Результат первой сессии — `plan.md` на 35 секций. Не код, а документ: стек, режимы, форматы файлов, примеры STATE.json, роадмап. Потом отдал план на ревью другой модели. Внешний взгляд нашёл реальные проблемы.

---

## Решения которые пришли через спор

**i18n в v0.1.** Казалось разумным — сразу на двух языках. Claude сам же предложил убрать: «зачем, если главная гипотеза — работает ли qwen14b на реальном проекте». Убрали в v0.4.

**review vs ask.** Два режима с нечёткой границей. После нескольких итераций пришли к одному критерию: `ask` — диалог, полные файлы не читаются; `review` — читает код, выдаёт структурированный отчёт. Внешняя модель предложила третий режим `inspect` как промежуточный — я отклонил. Три похожих режима сложнее одного чёткого разреза.

**LLM-summaries при init.** Первый вариант: `init` прогоняет каждый файл через LLM для INDEX. На локальной 14B это минуты. Исправление: по умолчанию AST-символы без LLM, LLM только при `--summarize`.

## Ключевые архитектурные решения

### Stateless между шагами

Главный инсайт: никакой истории разговора между LLM-вызовами. Каждый вызов — свежий промт:

```
[system.md] + [mode.md] + [stable_context] + [assistant: OK] + [dynamic_context]
```

`stable_context` меняется редко — ARCH.md, INDEX metadata, skill snippets. Сидит в KV-кэше.
`dynamic_context` меняется каждый шаг — задача, файлы, traceback, diff.

Вместо истории — step summaries в TASKS.md. После каждого шага: 1–3 строки что было сделано. Следующий шаг читает summary, не полную историю. Это и есть встроенный compact.

### Контекстный бюджет как первоклассная концепция

Не «постараемся уложиться», а явные лимиты и стратегии сжатия по приоритету:

```
1. file index      → только релевантные записи
2. code snippets   → обрезать до N строк + список символов
3. test output     → только traceback
4. git diff        → без context lines
→ всё равно не влезло → StopCondition(context_budget_exceeded)
```

`context_tokens` — автодетект из `/v1/models` API. Никаких хардкодных чисел. Все пороги (когда бар желтеет, когда предлагать compact) — в конфиге.

### Compact как семантический переход

Compact не только про «контекст кончается». Второй сценарий: переходишь из `ask` в `plan` после долгого обсуждения. Тащить весь чат не нужно — нужны решения. Агент суммаризирует диалог, суммари идёт в контекст планировщика. Планировщик не переспрашивает то что уже решили.

### Skills как Markdown-плагины

Новый язык или инструмент — не нужно менять код приложения. Создаёшь `.md` с YAML frontmatter:

```yaml
---
name: typescript
type: language
file_patterns: ["*.ts", "*.tsx"]
test_cmd: ["npx", "jest"]
symbol_pattern: "^(?:export\\s+)?(?:class|function)\\s+(\\w+)"
---
```

Агент сканирует директории при старте, приоритет: project > user > builtin. Команда `learn` генерирует skill через LLM из знаний модели или URL документации.

## TUI: от экранов к потоку

Первый вариант: отдельные Textual Screen-ы для каждой фазы. PatchScreen, PlanScreen, TestScreen. Это неправильно — ломает ощущение непрерывности.

Правильно: один экран, бесконечный скролл, inline-карточки в потоке. Вдохновение — Claude Code. Всё идёт в одну ленту, карточки появляются где нужно и становятся read-only после действия. Никаких попапов.

Начали с 8 типов карточек. После анализа что реально отличается — осталось 4:

| Тип | Что делает |
|---|---|
| `ToolCallCard` | все вызовы инструментов: read, search, git, apply, tests |
| `PlanCard` | список задач с прогрессом |
| `FormCard` | любая форма: QA-уточнения, настройки |
| `ChoiceCard` | бинарный выбор: resume, compact-offer |

`ToolCallCard` для Apply имеет два состояния: интерактивный (diff + [a]/[r]/[g]) и read-only (результат). Один виджет, меньше когнитивной нагрузки.

## Какую боль решаем

Рынок AI coding tools выглядит примерно так:

| Сегмент | Что есть |
|---|---|
| «магия в облаке» | Claude Code, Cursor, GitHub Copilot |
| «автономные агенты» | OpenHands, OpenDevin |
| «patch assistant» | Aider |
| «terminal + local» | куски через Ollama, LM Studio |

Почти нет инструмента с философией: локальная модель + маленький контекст + patch-oriented + supervised autonomy + TUI-first + engineering discipline. Это и есть ниша code-scalpel.

Но важнее — конкретная боль. У людей есть 7B–14B модели, но нет нормального coding workflow под них. Типичный опыт сейчас:

```
OpenCode/OpenHands
↓ контекст взорвался
↓ модель галлюцинирует
↓ агент переписал полпроекта
↓ всё сломалось
```

code-scalpel строит противоположный UX: не доверяй модели, ограничивай её, режь scope, показывай patch, тестируй быстро.

Аудитория — Linux/devops люди, локальные AI-хоббисты, privacy-first пользователи, embedded/infra/backend разработчики, те кто ненавидит web IDE, те у кого нет RTX 5090.

**Идеологическая цельность** — отдельный актив. Многие AI tools выглядят так: добавим MCP, добавим browser, добавим swarm, добавим memory graph. Философия code-scalpel простая: точечные изменения под слабые модели. Для open-source проекта это важно.

Главный враг сейчас — feature creep. Если v0.1 реально заработает (patch loop + qwen14b + pytest + diff preview) — проект уже имеет ценность сам по себе.

## Что вынес

**Документ перед кодом.** `plan.md` — не документация к коду, а инструмент мышления. Хорошо написанный план находит противоречия до того как они войдут в реализацию.

**Внешнее ревью работает.** Отдать план другой модели — не проверка грамматики, а реально другой угол зрения. Нашли то что пропустили вдвоём.

**Scope discipline — самое сложное.** Не «что добавить», а «что убрать». i18n в v0.1 казалась разумной пока не задали вопрос: зачем это для проверки главной гипотезы?

**AI не замена мышлению.** Claude предлагал, я отклонял, оба искали лучшее решение. Полезнее всего — когда модель задаёт неудобные вопросы, а не просто соглашается.
