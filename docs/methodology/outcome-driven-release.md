# Outcome-driven release

## Что это

Каждый релиз должен закрыть **один observable outcome** — конкретное
наблюдаемое состояние системы, которого до релиза не было. Acceptance
релиза = достигнут этот outcome или нет. Сколько фич landed, какой
test coverage, какие KPI выросли — не важно, если outcome не достигнут.

Альтернатива — **feature-driven release**: «landed фичи X, Y, Z,
релизим». Это работает для CRUD-задач, где outcome тривиально
выводится из списка фич (добавил кнопку → пользователь может на неё
нажать). Для систем с emergent behavior (ML, distributed systems,
agentic workflows) этого недостаточно: можно land'ить десять фич
поверх неработающего ядра и никогда этого не заметить.

## Когда применять

- **Сложная система с emergent behavior** — outcome не выводится из
  суммы фич. Например: «агент закрывает реальную задачу end-to-end»
  не складывается из «есть tool A» + «есть tool B».
- **Стохастический outcome** — нужны измерения, не unit-test. Например:
  «90% запросов на onboarding завершаются успехом».
- **Непонятный root-cause** — много возможных мест откуда блокер,
  нужен порядок приоритетов по данным.
- **Кросс-функциональная зависимость** — outcome требует чтобы N
  компонентов работали вместе.

## Когда **не** применять

- **Feature-driven достаточно** — CRUD admin, утилиты, простые API.
  Там outcome это и есть features.
- **Однозначное unit-testable поведение** — где `assert` проверяет
  тот outcome который ты хочешь.
- **Hot-fix критических багов** — там outcome очевиден («баг не
  воспроизводится»), нет нужды формализовать.

## Acceptance criterion — как формулировать

Хороший acceptance:
- **Numeric или binary**, не «улучшилось» — `≥1 task_solved`,
  `pass rate ≥ 0.7`, `p95 latency ≤ 200ms`.
- **Observable через автоматическую метрику** — не «выглядит лучше».
- **На интегрированной системе**, не на изолированных тестах.
  Capability-метрики (unit-test проходит) ≠ outcome-метрики
  (workflow завершился).
- **Один на релиз**. Если хочешь два — это два релиза.

Примеры:
- Плохо: «улучшен UX onboarding». (что значит «улучшен»?)
- Хорошо: «≥70% новых пользователей доходят до first-use в одной
  сессии, измеряется через session replay).
- Плохо: «agent умнее».
- Хорошо: «agent закрывает ≥1 задачу типа X в end-to-end probe-сценарии».

## Цикл итерации

```
1. Baseline measurement
   Прогон N≥3 для стохастических систем (один прогон — это шум).
   Записать distribution: mean, std, success rate.
   Это твоя точка отсчёта, к ней будут сравниваться все правки.

2. Reality-разбор данных (не нарративов)
   Открыть логи / артефакты / трассы прогонов. Что реально
   происходило? Не «вероятно блокер в X» — а «в run 1 на step 3
   произошло Y, в run 2 на step 5 — Z». Конкретные наблюдения.

3. Гипотеза о блокере
   Из наблюдений → одна гипотеза о том, что блокирует outcome.
   Не «вот тут вроде логично», а «вот это конкретное событие в
   данных не должно происходить». Должно покрываться unit-тестом
   ИЛИ N≥2 одинаковыми observations в данных.

4. Минимальная правка
   Закрыть конкретную гипотезу. Без захода в смежные улучшения.
   Прошёл unit-тест? — отлично. Прошёл код-ревью? — отлично.
   Это не достаточно. Достаточно только step 5.

5. Re-measurement
   N≥3 повторных прогонов на той же выборке. Сравнить с baseline.
   - Если distribution сдвинулся → правка работает, продолжить с
     новым baseline (step 1).
   - Если не сдвинулся → правка решает не ту проблему ИЛИ блокер
     где-то ещё. Откатить ИЛИ копать глубже, но не «оставим, пусть
     будет».
   - Если стало хуже — точно откатить.
```

Цикл повторяется пока acceptance не достигнут.

## Правила

1. **НИКОГДА не писать код раньше чем посмотрел данные.**
   Если ловишь себя за «вот тут вроде логично, сейчас напишу
   правку» — стоп. Открой логи / трассы / артефакты последних
   прогонов **сначала**. Опиши **из данных** что именно
   происходит. Сформулируй гипотезу с привязкой к конкретному
   observation. Только потом код.

   Это **самое нарушаемое** правило. Соблазн писать «логичные»
   правки огромен. Каждое нарушение этого правила = новая точка
   провала методики (см. антипаттерны 5-9 ниже). Если правка
   написана без observation — она с большой вероятностью решает
   не ту проблему, и ты увидишь это только когда правка не даст
   сдвига в outcome (если ещё **заметишь**, без re-measurement).

2. **Не land'ить правки без measurement**. Каждое изменение либо
   проходит N≥3 post-merge, либо имеет независимое обоснование
   через unit-тест. Иначе ловушка N=1 атрибуции.

3. **Не строить нарратив поверх случайных хороших прогонов**. Один
   удачный run на стохастической системе — это шум, не сигнал.
   N≥3 для любого утверждения «X лучше Y».

4. **Каждое observation проверять данными до построения теории**.
   Прежде чем сказать «я думаю блокер тут» — открой логи и убедись
   что блокер тут происходит хотя бы 2 раза независимо.

5. **Reflection после каждой итерации**. Что узнали о системе?
   Что узнали о методике? Какие гипотезы оказались неверны?

6. **Acceptance — единственный gate**. Не «landed N фич — релизим».
   Если acceptance не достигнут — версия не закрывается, даже
   если все фичи зелёные.

## Антипаттерны

Список ловушек, в которые легко попасть:

1. **Capability ≠ outcome.** Мерим что-то, что выглядит как outcome,
   но это input/capability метрика. «Unit-тесты проходят» ≠ «фича
   работает для пользователя». «Модель отвечает» ≠ «модель решает
   задачу».

2. **Acceptance pинятый поздно.** Outcome-метрика должна работать с
   v0.1, не с v0.N. Иначе ты узнаёшь что система не работает через
   полгода и N версий нагромождённой архитектуры сверху.

3. **Активность ≠ прогресс.** Особенно на стохастических системах:
   модель может тратить много токенов на полу-полезную работу не
   доводя до end-state. Если acceptance мерит активность (tool
   calls, tokens) — релиз выглядит успешным без реального outcome.

4. **Артефакт-сдвиг.** Документация / тесты / refactoring могут
   стать заменой реальной работы над outcome. Если на этой неделе
   нет коммитов которые двигают outcome-метрику — неделя не
   продуктивна, как бы хорошо ни шла документация.

5. **N=1 атрибуция.** Каждая точка измерения — это **одно**
   стохастическое решение системы. Нарратив «X лучше Y» требует
   distribution, не точки. Один прогон ничего не доказывает.

6. **Экстраполяция отрицательного вывода.** «На этом конкретном
   прогоне правка не помогла → правка бесполезна». Прогон мог
   быть в той области выборки, где правка не применима. Нужны
   данные на N≥3 разнообразных прогонах.

7. **Дырявая диагностическая инфраструктура.** Логи неполные,
   метрики собирают агрегаты, нет возможности воспроизвести.
   Любая выводы строится на этих данных. Прежде чем строить
   теорию — проверить, что данные полные.

8. **Атрибуция блокера не к той фазе pipeline'а.** В системе
   из N стадий блокер может быть на любой. Не предполагать «вот
   тут наверное» — проверить **каждую стадию** прогона.

9. **Постмортем без verification.** «Я понял почему упало» — это
   гипотеза, не вывод. Без re-прогона который её подтверждает,
   ты построил красивую теорию и можешь снова land'ить под
   неверную модель.

## Параллели с другими методиками

- **Definition of Done (Scrum)** — outcome-driven формализует DoD
  как обязательную метрику, не чек-лист.
- **A/B testing** — каждая правка измеряется как «изменила ли
  outcome», как варианты A/B измеряются на conversion.
- **SLO-based prioritization (SRE)** — фичи запрещены пока SLO
  не восстановлены. То же что «релиз не закрывается пока outcome
  не достигнут».
- **Test-driven development** — TDD требует тест до кода;
  outcome-driven требует **outcome-метрику** до кода. Похоже по
  идее, отличается уровнем (тест на unit vs метрика на систему).

## Стоимость

Outcome-driven сложнее feature-driven:

- **Сложнее планировать**: не знаешь сколько правок понадобится.
  v0.X может быть 2 PR'а или 12. Это нормально.
- **Сложнее коммуницировать**: нет списка фич для отчёта. «Что
  мы сделали в Q3?» — «закрыли два outcome'а». Звучит как меньше,
  чем «сделали 47 тикетов».
- **Легче ошибиться в acceptance**: если выбрали не ту метрику —
  оптимизируете не туда. Capability вместо outcome — самая частая
  ошибка.
- **Требует культуры measurement**. Без неё команда land'ит «по
  ощущениям».

Награда — гарантия что релиз реально полезен. На feature-driven
ты можешь land'ить десятки фич поверх неработающего ядра. На
outcome-driven этого не происходит по определению.
