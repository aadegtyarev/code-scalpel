# Scenario: a_diag_plan

## Project

Старт с **пустой fixture'ы** `mini_cli` (только pyproject.toml и
существующий код — но в этом сценарии используется как
«стартовая точка для расширения»). Цель — провести диалоговое
планирование: непрофессиональный юзер хочет улучшить
существующее CLI, не знает технических деталей.

В probe-runner'е fixture-проект всё ещё `mini_cli` (с рабочим
кодом и тестами), но сценарий просит scalpel'а **спланировать
расширение, не реализовывать**. То есть TASKS.md создаётся
с задачами «добавить X, Y, Z», но реальный код не пишется в
этом прогоне.

## User role

Я (Claude) — нетехнический заказчик, у которого есть идея.
Согласно `memory/user_tone_of_voice.md`: короткие реплики,
строчные буквы, опечатки оставляю, без вежливости-наполнителя.
На технические вопросы scalpel'а отвечаю «не знаю, реши сам» /
«просто чтобы работало».

## Цель

scalpel должен:
1. Поинтересоваться текущим состоянием проекта (запустить
   project_map или прочитать README/pyproject).
2. Уточнить пару ключевых вещей у юзера (но не больше 2-3 —
   мы наблюдаем за balance между «спросить» и «решить»).
3. Сгенерировать `.code-scalpel/TASKS.md` с осмысленным
   разбиением: задачи с Files / Test command / Acceptance.
4. Объяснить план в 2-3 фразах.

## Реальный запрос (то что я говорю первой репликой)

> хочу добавить в этот мини todo-cli возможность делать ярлыки
> на задачах. типа важное / срочное / личное. чтоб можно было
> фильтровать в list. как сделать?

## Success criteria (mechchecker → verdict.json)

| Критерий | Описание |
|---|---|
| `tasks_md_present` | `.code-scalpel/TASKS.md` существует после прогона |
| `tasks_count_ge_3` | в TASKS.md ≥ 3 задачи (по `## [_] T` заголовкам) |
| `tasks_have_required_fields` | каждая задача имеет хотя бы Files / Test command / Acceptance |
| `paths_valid` | все Files-пути относительные и не выходят за project root |

Плюс — я post-mortem ставлю `judge_score` 1-5 на «пошёл бы я по
этому плану в реализацию?» в `evaluation.md`.

## Что НЕ говорить

- Не предлагать структуру данных
- Не упоминать pydantic / dataclasses / sqlite — пусть scalpel
  сам решит
- Не давать список задач — это его работа

Если scalpel слишком долго спрашивает (>4 turn'ов чистых вопросов
без действий) — подталкиваю: «давай ты сам решай, что лучше».

## Зачем этот кейс

Жанр «диалог с непрофессионалом» — типичный сценарий ранней
итерации продукта, когда задачи приходят неструктурированными.
Измеряет:
- умеет ли 14b держать многоходовой диалог без потери темы
- умеет ли вытащить достаточно контекста из пары реплик
- генерирует ли валидный TASKS.md по нашему DSL

Контраст с `c_fix_bug` — там focused engineering, здесь — open
exploration.
